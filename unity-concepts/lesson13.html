1. What is Encapsulation? 🔒

Concept:

Encapsulation is the practice of hiding internal data and exposing only what's necessary through controlled interfaces. It's the third and final pillar of Object-Oriented Programming, and it protects your code from unwanted modification and makes debugging easier.

The Core Principle

Encapsulation Definition
ENCAPSULATION = Data Hiding + Controlled Access

Keep data PRIVATE/PROTECTED
Expose through PUBLIC METHODS/PROPERTIES

REAL-WORLD ANALOGY:
Your car engine:
- Internal parts are hidden (encapsulated)
- You interact through: steering wheel, pedals, buttons
- You don't need to know how engine works
- You can't accidentally break internal parts

PROGRAMMING:
Enemy health:
- Health variable is private
- You interact through: TakeDamage(), Heal(), GetHealth()
- You can't accidentally set health to negative
- You can't bypass game rules

🎯 Why Encapsulation Matters

  • Data Protection: Prevent accidental or malicious modification
  • Controlled Access: Decide who can read vs write data
  • Easier Debugging: Know exactly where data is modified
  • Flexibility: Change internal implementation without breaking other code
  • Validation: Enforce rules when data is modified
  • Cleaner Inspector: Hide internal variables from Unity Inspector

The Three Pillars Complete

OOP Pillars Recap
1. INHERITANCE (Lesson 11):
- Share code between classes
- Parent-child relationships
- Code reusability

2. POLYMORPHISM (Lesson 12):
- Same method, different behaviors
- virtual and override keywords
- Flexible, scalable systems

3. ENCAPSULATION (This Lesson):
- Hide internal data
- Control access with methods/properties
- Protect code integrity

TOGETHER: Professional, maintainable, scalable code!
💡 Philosophy: "Make everything as private as possible, then selectively expose what's needed." This is the golden rule of encapsulation and the key to clean, professional code!

2. The Problem: Why Not Make Everything Public? ⚠️

Concept:

The easiest solution to accessing variables from other scripts is making them public. But this creates massive problems in real projects. Let's see why this "easy" solution is actually a disaster waiting to happen.

The Scenario

Needing Data from Another Script
// Enemy.cs
public class Enemy : MonoBehaviour
{
    [SerializeField] protected string enemyName = "Skeleton";
    [SerializeField] protected int maxHealth = 100;
    protected int currentHealth;
}

// PlayerController.cs
public class PlayerController : MonoBehaviour
{
    private void OnTriggerEnter2D(Collider2D collision)
    {
        Enemy enemy = collision.GetComponent<Enemy>();
        if (enemy != null)
        {
            enemy.TakeDamage(10);

            // Want to log which enemy we damaged
            Debug.Log("I damaged enemy: " + ???);
            // ❌ Can't access enemyName - it's protected!
        }
    }
}

The Tempting "Solution": Make It Public

❌ BAD: Public Everything
// Enemy.cs

{
    // Made public to access from Player

    public int maxHealth = 100;
    public int currentHealth;

  • Not Professional: Real projects would be unmaintainable chaos

3. Solution 1: Getter Methods 📤

Concept:

A getter method is a public method that returns a protected/private variable. The variable stays protected (can't be modified externally), but other scripts can read its value through the method. This is classic encapsulation.

Creating a Getter Method

Enemy.cs - Encapsulated with Getter
using UnityEngine;

public class Enemy : MonoBehaviour
{
    // Protected - only this class and children can access
    [SerializeField] protected string enemyName = "Enemy";
    [SerializeField] protected int maxHealth = 100;
    protected int currentHealth;

    // Public getter method - allows reading from outside
    public string GetEnemyName()
    {
        return enemyName;
    }

    // Another getter for health
    public int GetCurrentHealth()
    {
        return currentHealth;
    }

    protected virtual void Awake()
    {
        currentHealth = maxHealth;
    }

    public virtual void TakeDamage(int damage)
    {
        currentHealth -= damage;
        if (currentHealth <= 0)
        {
            Die();
        }
    }

    protected virtual void Die()
    {
        Debug.Log(enemyName + " dies!");
        Destroy(gameObject);
    }
}

🎯 How Getter Methods Work

  • Method Returns Value: GetEnemyName() returns the enemyName string
  • Variable Stays Protected: Can't modify enemyName from outside
  • Read-Only Access: Other scripts can read but not write
  • Naming Convention: Get[VariableName]() is standard (GetHealth, GetSpeed)
  • Return Type Matches: Method return type = variable type

Using the Getter in Player Script

PlayerController.cs
using UnityEngine;

public class PlayerController : MonoBehaviour
{
    private void OnTriggerEnter2D(Collider2D collision)
    {
        // Get the Enemy component
        Enemy enemyScript = collision.GetComponent<Enemy>();

        if (enemyScript != null)
        {
            // Damage the enemy
            enemyScript.TakeDamage(10);

            // Get enemy name using getter method
            string name = enemyScript.GetEnemyName();
            Debug.Log("I damaged enemy: " + name);

            // Can also use directly in string
            Debug.Log("I damaged enemy: " + enemyScript.GetEnemyName());
        }
    }
}

What You Can and Can't Do

Access Control
Enemy enemyScript = collision.GetComponent<Enemy>();

// ✅ CAN DO: Read value through getter
string name = enemyScript.GetEnemyName();
int health = enemyScript.GetCurrentHealth();
Debug.Log("Enemy: " + name + " has " + health + " HP");

// ❌ CAN'T DO: Access variable directly
// string name = enemyScript.enemyName; // ERROR: inaccessible

// ❌ CAN'T DO: Modify the value
// enemyScript.enemyName = "NewName"; // ERROR: inaccessible
// You can't assign to a method: enemyScript.GetEnemyName() = "NewName";

// ✅ CAN DO: Use public methods to modify safely
enemyScript.TakeDamage(10); // Uses controlled method

RESULT: Read-only access from outside, full control inside!
💼 Professional Pattern: Getter methods are the classic OOP approach. They're clear, explicit, and easy to understand. Many professional codebases use this pattern extensively!

4. Solution 2: C# Properties (Get/Set) 🎛️

Concept:

C# properties are a more elegant way to encapsulate data. They look like variables but act like methods, allowing you to control read and write access. The syntax is cleaner than getter methods and is the modern C# approach.

Basic Property Syntax

Properties vs Getter Methods
// OLD WAY: Getter method
private string enemyName = "Skeleton";

public string GetEnemyName()
{
    return enemyName;
}

// NEW WAY: Property
private string _enemyName = "Skeleton";

public string EnemyName
{
    get { return _enemyName; }
    set { _enemyName = value; }
}

// USAGE COMPARISON:
// Getter method:
string name = enemy.GetEnemyName();

// Property:
string name = enemy.EnemyName; // Cleaner!

🎯 Property Basics

  • get Accessor: Returns the value (like a getter method)
  • set Accessor: Assigns a value (value keyword is the new value)
  • Backing Field: Usually private variable with underscore (_enemyName)
  • Naming Convention: Property name is PascalCase (EnemyName, not enemyName)
  • Looks Like Variable: Used like variable but runs method code

Full Property Example

Enemy.cs with Properties
using UnityEngine;

public class Enemy : MonoBehaviour
{
    // Backing fields (private)
    [SerializeField] private string _enemyName = "Enemy";
    [SerializeField] private int _maxHealth = 100;
    private int _currentHealth;

    // Property with full get/set
    public string EnemyName
    {
        get { return _enemyName; }
        set { _enemyName = value; }
    }

    // Property with validation in setter
    public int CurrentHealth
    {
        get { return _currentHealth; }
        set
        {
            // Validation: health can't go below 0
            _currentHealth = Mathf.Clamp(value, 0, _maxHealth);

            // Automatically die if health reaches 0
            if (_currentHealth <= 0)
            {
                Die();
            }
        }
    }

    private void Awake()
    {
        _currentHealth = _maxHealth;
    }

    protected virtual void Die()
    {
        Debug.Log(_enemyName + " dies!");
        Destroy(gameObject);
    }
}

Using Properties

PlayerController.cs
// Reading property (calls get)
string name = enemyScript.EnemyName;
int health = enemyScript.CurrentHealth;
Debug.Log(name + " has " + health + " HP");

// Writing property (calls set)
enemyScript.CurrentHealth = 50; // Setter runs with validation
enemyScript.EnemyName = "Elite Skeleton";

// Looks like variable access, but methods run behind the scenes!

// Operations work too
enemyScript.CurrentHealth -= 10; // get, subtract, then set
// Equivalent to:
// enemyScript.CurrentHealth = enemyScript.CurrentHealth - 10;
✨ Power of Properties: Properties can run complex logic in get/set. Clamp health to valid range, trigger events, validate input—all invisible to the caller! This is encapsulation at its finest.

5. The Magic Pattern: "public get; private set;" 🪄

Concept:

The most common property pattern in Unity: public get; private set; This creates a property that anyone can read, but only the owning class can modify. It's the perfect balance of accessibility and protection.

The Auto-Property Syntax

Short and Sweet
// LONG WAY: Manual backing field
private string _enemyName = "Skeleton";
public string EnemyName
{
    get { return _enemyName; }
    private set { _enemyName = value; }
}

// SHORT WAY: Auto-property (C# does backing field automatically)
public string EnemyName { get; private set; } = "Skeleton";

// Even shorter (set default in constructor/Awake)
public string EnemyName { get; private set; }

// Multiple examples
public int CurrentHealth { get; private set; }
public float MoveSpeed { get; private set; } = 3f;
public bool IsDead { get; private set; } = false;

🎯 How It Works

  • public get: Anyone can read the property
  • private set: Only this class can modify the property
  • Auto-Property: C# creates backing field automatically (you don't see it)
  • Read-Only to Outside: Other scripts see it as read-only
  • Full Control Inside: This class can set it freely

Complete Example

Enemy.cs - Modern Pattern
using UnityEngine;

public class Enemy : MonoBehaviour
{
    // Public read, private write properties
    public string EnemyName { get; private set; } = "Enemy";
    public int MaxHealth { get; private set; } = 100;
    public int CurrentHealth { get; private set; }
    public bool IsDead { get; private set; } = false;

    // Still need SerializeField for Inspector visibility
    [SerializeField] private string _inspectorName = "Skeleton";
    [SerializeField] private int _inspectorMaxHealth = 100;

    protected virtual void Awake()
    {
        // Set property values from inspector fields
        EnemyName = _inspectorName;
        MaxHealth = _inspectorMaxHealth;
        CurrentHealth = MaxHealth;
    }

    public virtual void TakeDamage(int damage)
    {
        CurrentHealth -= damage; // ✅ Can modify inside this class
        CurrentHealth = Mathf.Max(CurrentHealth, 0);

        if (CurrentHealth <= 0)
        {
            Die();
        }
    }

    protected virtual void Die()
    {
        IsDead = true; // ✅ Can set inside this class
        Debug.Log(EnemyName + " dies!");
        Destroy(gameObject);
    }
}

Using Read-Only Properties

PlayerController.cs
Enemy enemyScript = collision.GetComponent<Enemy>();

// ✅ CAN READ: Public get allows reading
string name = enemyScript.EnemyName;
int health = enemyScript.CurrentHealth;
bool dead = enemyScript.IsDead;

Debug.Log("I damaged enemy: " + name);
Debug.Log("Remaining health: " + health);

// ❌ CANNOT WRITE: Private set prevents modification
// enemyScript.EnemyName = "NewName"; // ERROR: inaccessible setter
// enemyScript.CurrentHealth = 999; // ERROR: inaccessible setter
// enemyScript.IsDead = true; // ERROR: inaccessible setter

// ✅ Must use public methods for controlled modification
enemyScript.TakeDamage(10);
💼 Industry Standard: "public get; private set;" is the most common pattern in professional Unity projects. It's clean, clear, and protects data while allowing read access. Use this by default!

6. When to Use Each Approach 🤔

Concept:

You now know three ways to expose data: getter methods, full properties, and auto-properties with "get; private set;". Each has its place. Here's when to use which approach.

Method 1: Getter Methods

✅ Use Getter Methods When:

  • Complex Calculation: GetDamageOutput() calculates damage with formulas
  • Method Indicates Action: "Get" suggests something happens
  • Multiple Parameters: GetHealthPercentage(bool includeShield)
  • Working with Legacy Code: Existing codebase uses getter methods
  • Personal Preference: Some teams prefer explicit methods
Getter Method Example
public float GetHealthPercentage()
{
    return (float)currentHealth / maxHealth * 100f;
}

public int GetTotalDamage()
{
    return baseDamage + weaponDamage + buffDamage;
}

public string GetDisplayName()
{
    return isElite ? "Elite " + enemyName : enemyName;
}

Method 2: Full Properties (get + set)

✅ Use Full Properties When:

  • Validation Needed: Clamp, check range, enforce rules in setter
  • Side Effects: Setting value triggers events or updates UI
  • Computed Values: Get calculates, set updates backing fields
  • Public Read/Write: Both get and set should be public
Full Property Example
private int _currentHealth;

public int CurrentHealth
{
    get { return _currentHealth; }
    set
    {
        // Validation
        _currentHealth = Mathf.Clamp(value, 0, maxHealth);

        // Side effect: update UI
        UpdateHealthBar();

        // Side effect: check death
        if (_currentHealth <= 0)
        {
            Die();
        }
    }
}

Method 3: Auto-Properties (get; private set;)

✅ Use Auto-Properties When:

  • Simple Data Storage: Just storing a value, no computation
  • Read-Only to Outside: Public read, private write pattern
  • Clean Syntax Preferred: Shorter, more readable code
  • Modern C# Style: Industry standard for Unity
  • No Validation Needed: Direct value storage
Auto-Property Example
public string EnemyName { get; private set; }
public int MaxHealth { get; private set; } = 100;
public bool IsDead { get; private set; } = false;
public float MoveSpeed { get; private set; } = 3f;

Decision Tree

Which Approach to Use?
DECISION TREE:

Need calculation or complex logic?
→ YES: Use Getter Method (GetHealthPercentage())

Need validation or side effects when setting?
→ YES: Use Full Property with custom get/set

Need public read, private write, simple storage?
→ YES: Use Auto-Property { get; private set; }

Need public read AND write with validation?
→ YES: Use Full Property with public get/set

Need public read AND write, no validation?
→ Consider: Do you REALLY need public write?
→ Usually better: Keep private, use method like SetHealth(int value)

DEFAULT CHOICE:
When in doubt → Auto-Property { get; private set; }
🎯 Professional Mix: Real projects use ALL three approaches in different places. Auto-properties for simple data, full properties for validated data, getter methods for calculations. Use the right tool for the job!

7. Debugging Benefits of Encapsulation 🔍

Concept:

One of the biggest benefits of encapsulation is easier debugging. When data can only be modified through specific methods or setters, you know exactly where to look when something goes wrong.

The Public Variable Nightmare

❌ Public Variable Debugging Nightmare
// Enemy.cs
public int currentHealth = 100;

BUG REPORT: "Enemy has negative health!"

WHERE IS IT MODIFIED?
- Could be PlayerController.cs (line 45)
- Could be TrapScript.cs (line 92)
- Could be PoisonEffect.cs (line 31)
- Could be BossAttack.cs (line 78)
- Could be CheatConsole.cs (line 104)
- Could be PowerUpManager.cs (line 56)
- Could be 30 other scripts!

DEBUGGING PROCESS:
1. Search entire project for "currentHealth"
2. Find 147 references
3. Check each one manually
4. Add Debug.Log everywhere
5. Spend 2 hours finding the bug
6. It was a typo in obscure script you forgot about

CONCLUSION: Nightmare!

The Encapsulated Solution

✅ Encapsulated Debugging Paradise
// Enemy.cs
public int CurrentHealth { get; private set; } = 100;

public void TakeDamage(int damage)
{
    Debug.Log($"TakeDamage called: {damage} damage");
    CurrentHealth -= damage;

    if (CurrentHealth < 0)
    {
        Debug.LogError($"Health went negative! CurrentHealth: {CurrentHealth}");
    }
}

BUG REPORT: "Enemy has negative health!"

WHERE IS IT MODIFIED?
- Only in TakeDamage() method
- That's it. ONE place.

DEBUGGING PROCESS:
1. Put breakpoint in TakeDamage()
2. See which call makes health negative
3. Fix the bug in 5 minutes

CONCLUSION: Easy!

Adding Debugging to Properties

Debug-Friendly Property
private int _currentHealth;

public int CurrentHealth
{
    get
    {
        return _currentHealth;
    }
    private set
    {
        // Log every change for debugging
        Debug.Log($"Health changed: {_currentHealth} → {value}");

        // Catch invalid values
        if (value < 0)
        {
            Debug.LogError($"Attempting to set negative health: {value}");
            Debug.LogError($"Stack trace: {System.Environment.StackTrace}");
        }

        _currentHealth = value;
    }
}

// Now you see EVERY health change with stack trace!
// Turn on/off with bool flag for release builds

🎯 Encapsulation Debugging Benefits

  • Single Point of Modification: Only one place to check
  • Easy Breakpoints: Put breakpoint in setter/method
  • Logging: Add Debug.Log in one place
  • Validation: Catch invalid values before they cause problems
  • Stack Traces: Know exactly what called the setter
  • Quick Fixes: Find and fix bugs in minutes, not hours
🚨 Real Experience: Senior developers spend 70% of time debugging. Encapsulation can reduce debugging time by 80%. That's the difference between 1 hour and 5 minutes to find a bug. Worth it!

8. Encapsulation with [SerializeField] 🎨

Concept:

Unity's [SerializeField] attribute lets you show private/protected variables in the Inspector without making them public. This maintains encapsulation while allowing designers to tune values—the best of both worlds!

The Problem with Properties in Inspector

❌ Properties Don't Show in Inspector
// This WON'T show in Inspector
public int MaxHealth { get; private set; } = 100;

// Neither will this
public string EnemyName { get; private set; } = "Skeleton";

WHY?
Unity only serializes FIELDS (variables), not PROPERTIES.
Properties are technically methods, not data storage.

PROBLEM:
Can't tune MaxHealth per enemy in Inspector!

The Solution: SerializeField + Private Field + Property

✅ Inspector + Encapsulation Pattern
using UnityEngine;

public class Enemy : MonoBehaviour
{
    // Private field - visible in Inspector via SerializeField
    [SerializeField] private string _enemyName = "Skeleton";
    [SerializeField] private int _maxHealth = 100;

    // Public read-only properties for code access
    public string EnemyName { get; private set; }
    public int MaxHealth { get; private set; }
    public int CurrentHealth { get; private set; }

    private void Awake()
    {
        // Copy Inspector values to properties
        EnemyName = _enemyName;
        MaxHealth = _maxHealth;
        CurrentHealth = MaxHealth;
    }
}

RESULT:
✅ Inspector shows _enemyName and _maxHealth (tunable)
✅ Code uses EnemyName and MaxHealth properties (read-only)
✅ Encapsulation maintained
✅ Designer-friendly

Alternative: Getter Method Pattern

Simpler Pattern for Getters
public class Enemy : MonoBehaviour
{
    // SerializeField + protected = Inspector visible + child accessible
    [SerializeField] protected string enemyName = "Skeleton";
    [SerializeField] protected int maxHealth = 100;

    // Public getter methods
    public string GetEnemyName() => enemyName;
    public int GetMaxHealth() => maxHealth;
}

// Usage in other scripts:
string name = enemy.GetEnemyName();
int health = enemy.GetMaxHealth();

PROS:
✅ Simpler - fewer variables
✅ Inspector-friendly
✅ Child classes can access protected fields

CONS:
❌ Method call syntax (parentheses)
❌ Less "modern" than properties

🎯 SerializeField Best Practices

  • Use Underscore: Private fields with underscore (_maxHealth) distinguish from properties
  • Set in Awake: Copy Inspector values to properties in Awake()
  • Protected OK: [SerializeField] protected works for child class access
  • Header Attribute: Use [Header("Stats")] to organize Inspector
  • Tooltip Attribute: Use [Tooltip("text")] to add descriptions

Full Professional Example

Production-Ready Pattern
using UnityEngine;

public class Enemy : MonoBehaviour
{
    [Header("Identity")]
    [SerializeField] private string _enemyName = "Skeleton";
    [SerializeField] private Sprite _enemyIcon;

    [Header("Stats")]
    [SerializeField] [Tooltip("Maximum health points")]
    private int _maxHealth = 100;

    [SerializeField] [Range(1f, 10f)]
    private float _moveSpeed = 3f;

    // Public properties (code access)
    public string EnemyName { get; private set; }
    public int MaxHealth { get; private set; }
    public int CurrentHealth { get; private set; }
    public float MoveSpeed { get; private set; }

    private void Awake()
    {
        EnemyName = _enemyName;
        MaxHealth = _maxHealth;
        CurrentHealth = MaxHealth;
        MoveSpeed = _moveSpeed;
    }
}

INSPECTOR RESULT:
Clean, organized sections with tooltips and sliders
Designers can tune without touching code

CODE RESULT:
enemy.EnemyName - clean property access
enemy.CurrentHealth - read-only from outside
🎨 Team Workflow: This pattern lets programmers protect code while giving designers full Inspector control. Perfect for team projects where non-programmers need to tune values!

9. Real-World Testing Example 🎮

Concept:

Let's put everything together with a complete, working example. We'll create an Enemy with encapsulated data and a Player that damages enemies while accessing their protected information properly.

Complete Enemy Script

Enemy.cs - Fully Encapsulated
using UnityEngine;

public class Enemy : MonoBehaviour
{
    // SerializeField for Inspector, private for encapsulation
    [SerializeField] private string _enemyName = "Skeleton";
    [SerializeField] private int _maxHealth = 100;

    // Public get, private set properties
    public string EnemyName { get; private set; }
    public int MaxHealth { get; private set; }
    public int CurrentHealth { get; private set; }
    public bool IsDead { get; private set; } = false;

    protected virtual void Awake()
    {
        EnemyName = _enemyName;
        MaxHealth = _maxHealth;
        CurrentHealth = MaxHealth;
    }

    public virtual void TakeDamage(int damage)
    {
        if (IsDead) return;

        CurrentHealth -= damage;
        CurrentHealth = Mathf.Max(CurrentHealth, 0);

        Debug.Log($"{EnemyName} took {damage} damage! Health: {CurrentHealth}/{MaxHealth}");

        if (CurrentHealth <= 0)
        {
            Die();
        }
    }

    protected virtual void Die()
    {
        IsDead = true;
        Debug.Log($"{EnemyName} dies!");
        Destroy(gameObject);
    }
}

Player Script Using Encapsulated Data

PlayerController.cs
using UnityEngine;

public class PlayerController : MonoBehaviour
{
    [SerializeField] private int attackDamage = 10;

    private void OnTriggerEnter2D(Collider2D collision)
    {
        // Try to get Enemy component
        Enemy enemyScript = collision.GetComponent<Enemy>();

        if (enemyScript != null)
        {
            // Damage the enemy (public method)
            enemyScript.TakeDamage(attackDamage);

            // Access encapsulated data (read-only properties)
            string name = enemyScript.EnemyName;
            int health = enemyScript.CurrentHealth;
            bool dead = enemyScript.IsDead;

            // Log what happened
            if (!dead)
            {
                Debug.Log($"I damaged enemy: {name} (Remaining HP: {health})");
            }
            else
            {
                Debug.Log($"I killed enemy: {name}!");
            }
        }
    }
}

// INHERITANCE: Base class with shared functionality
public class Enemy : MonoBehaviour
{
    // ENCAPSULATION: Protected data with public properties
    [SerializeField] private string _enemyName = "Enemy";
    public string EnemyName { get; private set; }

    [SerializeField] private int _maxHealth = 100;
    public int CurrentHealth { get; private set; }

    // POLYMORPHISM: Virtual method for child customization
    protected virtual void Attack()
    {
        Debug.Log(EnemyName + " attacks!");
    }

    // ENCAPSULATION: Controlled access through public method
    public void TakeDamage(int damage)
    {
        CurrentHealth -= damage;
        // Validation and side effects here
    }
}

// INHERITANCE: Child class inherits from Enemy
public class EnemyGoblin : Enemy
{
    // POLYMORPHISM: Override parent behavior
    protected override void Attack()
    {
        base.Attack(); // Parent behavior
        StealGold(); // Unique behavior
    }

    // ENCAPSULATION: Private helper method
    private void StealGold()
    {
        // Access parent's encapsulated data
        Debug.Log(EnemyName + " steals gold!");
    }
}
