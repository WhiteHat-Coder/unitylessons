1. Creating a Movement Animation 🎬

Concept:

A movement animation (also called run or walk animation) plays when your character is moving. It creates the illusion of walking or running by cycling through sprite frames.

Creating the Animation:

📋 Steps:

  1. Select your Graphics GameObject (the child with Animator)
  2. Open Window → Animation → Animation
  3. Click the dropdown next to "playerIdle" animation
  4. Select Create New Clip
  5. Save it in your Animations folder as "playerMove"

Adding Movement Sprites:

Selecting Run/Walk Frames
1. In Project window, expand your sprite sheet
2. Find the movement/run frames (usually after idle frames)
3. Select 6-8 consecutive run frames
Example: warrior_6 through warrior_13
4. Drag them into the Animation timeline
5. Set Sample Rate to 12

🎨 Sample Rate Guidelines:

  • Idle: 6-10 FPS (slow and calm)
  • Walk: 10-12 FPS (moderate)
  • Run: 12-15 FPS (energetic)
  • Attack: 15-24 FPS (fast and snappy)
💡 Tip: Preview your animation by clicking the Play button in the Animation window (not the main Play Mode button). Adjust the sample rate until it looks natural!

2. Understanding the Animator Window 🎭

Concept:

The Animator window shows your animation states as boxes and lets you create transitions between them. Think of it as a flowchart for your character's animations.

Opening the Animator:

📋 Steps:

  1. Open Window → Animation → Animator
  2. Select your Graphics GameObject
  3. You'll see your animation states (playerIdle, playerMove)
  4. Orange state = Default/Entry state

What You See:

Animator View
┌──────────┐
│ Entry │──→ ┌─────────────┐
└──────────┘ │ playerIdle │ ← Orange (default)
└─────────────┘

┌─────────────┐
│ playerMove │ ← Gray (not default)
└─────────────┘
🎯 Goal: We need to create transitions so the character can switch from Idle to Move and back based on whether they're moving or standing still.

3. Animator Parameters 🎛️

Concept:

Parameters are variables in the Animator that control when transitions happen. You can set these from your code to trigger different animations.

Types of Parameters:

Parameter Types
Float: Decimal number (e.g., speed: 0.0 to 5.5)
  Use for: Blend trees, gradual transitions

Int: Whole number (e.g., comboIndex: 0, 1, 2)
  Use for: Attack combos, weapon types

Bool: True or False (e.g., isMoving: true/false)
  Use for: Simple on/off states

Trigger: One-time signal (automatically resets)
  Use for: One-shot actions like attacks, jumps

Creating a Boolean Parameter:

📋 Steps:

  1. In Animator window, find the Parameters tab (left side)
  2. Click the + button
  3. Select Bool
  4. Name it exactly: isMoving
  5. Leave it unchecked (false by default)
⚠️ Important: Parameter names are case-sensitive! isMoving is different from IsMoving. Be consistent!

4. Creating Transitions Between States 🔄

Concept:

A transition is an arrow connecting two animation states. It defines when and how the Animator should switch from one animation to another.

Creating Idle → Move Transition:

📋 Steps:

  1. In Animator window, right-click on playerIdle
  2. Select Make Transition
  3. Click on playerMove to create the arrow
  4. Click the arrow (transition) to select it
  5. In Inspector, find Conditions
  6. Click + to add condition
  7. Set: isMoving = true

Creating Move → Idle Transition:

📋 Steps:

  1. Right-click on playerMove
  2. Select Make Transition
  3. Click on playerIdle
  4. Select the new arrow
  5. Add condition: isMoving = false
Transition Flow
┌─────────────┐ ┌─────────────┐
│ playerIdle │──isMoving=true───→ │ playerMove │
│ │←──isMoving=false── │ │
└─────────────┘ └─────────────┘
🧪 Testing: Enter Play Mode, select Graphics GameObject, and manually check/uncheck isMoving in the Animator window to test transitions!

5. Exit Time & Transition Duration ⏱️

Concept:

Exit Time makes the Animator wait for the current animation to finish before transitioning. Transition Duration controls how long the blend between animations takes.

The Problem with Defaults:

Default Behavior (Feels Sluggish)
Player presses movement key → Delay... → Character starts running
Player releases key → Delay... → Character stops

Why?
- Has Exit Time: ✅ (waits for animation to finish)
- Transition Duration: 0.25s (slow blend)

Fixing for Instant Response:

📋 For BOTH Transitions (Idle→Move AND Move→Idle):

  1. Click the transition arrow
  2. In Inspector, uncheck "Has Exit Time"
  3. Set Transition Duration to 0
  4. Repeat for the other transition
After Optimization
Player presses movement key → Character runs IMMEDIATELY ✅
Player releases key → Character stops IMMEDIATELY ✅

Settings:
- Has Exit Time: ❌ (instant transition)
- Transition Duration: 0 (no blend)

⚙️ When to Use Exit Time:

  • Enable for: Attack animations, death animations (let them finish)
  • Disable for: Movement, idle (need instant response)
💡 Pro Tip: Movement animations should feel responsive! Players expect immediate feedback when they press a key.

6. Accessing the Animator from Code 💻

Concept:

To control animations from code, you need a reference to the Animator component. Since it's on the child Graphics GameObject, we use GetComponentInChildren.

Declaring the Animator Variable:

In Your PlayerController Script
public class PlayerController : MonoBehaviour
{
    private Rigidbody2D rb;
    private Animator anim; ← Add this
    
    // ... other variables

Getting the Component:

In Awake() Method
void Awake()
{
    rb = GetComponent<Rigidbody2D>();
    anim = GetComponentInChildren<Animator>();
}

GetComponent vs GetComponentInChildren:

🔍 Understanding the Difference:

  • GetComponent<T>(): Searches ONLY on the same GameObject
  • GetComponentInChildren<T>(): Searches GameObject AND all children
  • Returns the FIRST matching component it finds
Your GameObject Structure
Player (← Script is here)
  ├── Rigidbody2D (same GameObject, use GetComponent)
  ├── Collider2D
  └── Graphics (Child GameObject)
       ├── SpriteRenderer
       └── Animator (child, use GetComponentInChildren)
💡 Remember: GetComponentInChildren searches the GameObject itself first, then its children. So even if Animator was on the parent, this would still work!

7. Local vs Global Variables 📦

Concept:

Global variables (fields) are declared at class level and accessible anywhere in the class. Local variables are declared inside methods and only exist within that method.

Global Variables:

Class-Level (Global)
public class PlayerController : MonoBehaviour
{
    private Animator anim; ← Global: accessible everywhere
    private Rigidbody2D rb; ← Global: accessible everywhere

    void Update() { /* can use anim and rb */ }
    void FixedUpdate() { /* can use anim and rb */ }
}

Local Variables:

Method-Level (Local)
void HandleAnimations()
{
    bool isMoving = rb.linearVelocity.x != 0; ← Local
    // isMoving only exists inside this method
}

void Update()
{
    // CANNOT use isMoving here! It doesn't exist outside HandleAnimations()
}

When to Use Each:

🎯 Best Practices:

  • Global: Components, settings, data needed by multiple methods
  • Local: Temporary calculations, data used only in one method
  • Why? Local variables save memory and prevent bugs from unintended access
💡 Rule of Thumb: Make variables as local as possible. Only make them global if multiple methods need access!

8. Detecting Movement with Velocity 🏃

Concept:

Velocity measures how fast an object is moving and in what direction. If velocity.x is 0, the player is standing still. If it's not 0, they're moving!

Checking Velocity in Play Mode:

🔬 Experiment:

  1. Enter Play Mode
  2. Select your Player GameObject
  3. In Inspector, expand Rigidbody2D
  4. Look at Info → Velocity
  5. Standing still: X = 0
  6. Moving left: X = negative (e.g., -3.5)
  7. Moving right: X = positive (e.g., 3.5)

Using Velocity to Detect Movement:

The Logic
// If velocity.x is NOT zero, player is moving
bool isMoving = rb.linearVelocity.x != 0;

Examples:
- velocity.x = 0 → isMoving = false (standing)
- velocity.x = 3.5 → isMoving = true (moving right)
- velocity.x = -2.8 → isMoving = true (moving left)

💡 Why This Works:

  • We don't care about direction (left/right), only IF they're moving
  • != 0 checks if velocity is anything other than zero
  • This automatically returns true or false
🎯 Alternative Methods: You could also check Input.GetKey(), but velocity is more reliable because it reflects actual physics movement, not just key presses!

9. Quick Boolean Assignment ⚡

Concept:

Instead of writing an if-else statement, you can assign a boolean directly from a condition. This is cleaner and more efficient!

The Long Way (Redundant):

Using If-Else (Not Recommended)
bool isMoving;

if (rb.linearVelocity.x != 0)
{
    isMoving = true;
}
else
{
    isMoving = false;
}

// 9 lines of code! 😰

The Short Way (Recommended):

Direct Assignment (Clean!)
bool isMoving = rb.linearVelocity.x != 0;

// 1 line of code! ✨
// If condition is true, isMoving = true
// If condition is false, isMoving = false

How It Works:

🧠 Understanding the Logic:

  • rb.linearVelocity.x != 0 evaluates to true or false
  • That boolean result is directly assigned to isMoving
  • No if-else needed because the condition already returns a boolean!
More Examples
// Is player grounded?
bool isGrounded = Physics2D.Raycast(transform.position, Vector2.down, 0.1f);

// Is player jumping?
bool isJumping = rb.linearVelocity.y > 0;

// Is health low?
bool isLowHealth = health < 30;
💡 Pro Tip: Whenever you write if (condition) x = true; else x = false;, replace it with x = condition;!

10. Controlling Animations with SetBool 🎮

Concept:

SetBool() is a method that changes a boolean parameter in the Animator, triggering transitions based on the conditions you set up.

The SetBool Method:

Syntax
animator.SetBool(string parameterName, bool value);

// parameterName: Must match EXACTLY what you named it in Animator
// value: true or false

Creating the HandleAnimations Method:

Complete Method
void HandleAnimations()
{
    // Detect if player is moving
    bool isMoving = rb.linearVelocity.x != 0;

    // Update the Animator parameter
    anim.SetBool("isMoving", isMoving);
}

Calling It Every Frame:

In Update() Method
void Update()
{
    HandleMovement();
    HandleJump();
    HandleAnimations(); ← Add this
}

Parameter Name Matching:

⚠️ Critical Points:

  • The string "isMoving" MUST match the Animator parameter name exactly
  • Case-sensitive: "isMoving" ≠ "IsMoving"
  • Pro Tip: Copy the parameter name from Animator window to avoid typos!
Complete Code Flow
1. Player presses A/D → Velocity changes
2. Update() calls HandleAnimations()
3. HandleAnimations() checks velocity.x
4. isMoving = true (if moving) or false (if stopped)
5. SetBool updates Animator parameter
6. Animator checks transition conditions
7. Animation switches (Idle ↔ Move)
🐛 Debugging: If animations don't work, check: (1) Parameter name spelling, (2) Animator is assigned, (3) Transitions have correct conditions!

11. Complete Code Implementation 📝

Concept:

Let's put everything together and see the complete code for controlling animations based on player movement.

Full PlayerController with Animations:

PlayerController.cs
using UnityEngine;

public class PlayerController : MonoBehaviour
{
    [SerializeField] private float moveSpeed = 5f;
    [SerializeField] private float jumpForce = 10f;

    private Rigidbody2D rb;
    private Animator anim;

    void Awake()
    {
        rb = GetComponent<Rigidbody2D>();
        anim = GetComponentInChildren<Animator>();
    }

    void Update()
    {
        HandleMovement();
        HandleJump();
        HandleAnimations();
    }

    void HandleMovement()
    {
        float horizontalInput = Input.GetAxisRaw("Horizontal");
        rb.linearVelocity = new Vector2(horizontalInput * moveSpeed, rb.linearVelocity.y);
    }

    void HandleJump()
    {
        if (Input.GetKeyDown(KeyCode.Space))
        {
            rb.linearVelocity = new Vector2(rb.linearVelocity.x, jumpForce);
        }
    }

    void HandleAnimations()
    {
        // Detect movement using velocity
        bool isMoving = rb.linearVelocity.x != 0;

        // Update Animator parameter
        anim.SetBool("isMoving", isMoving);
    }
}

✨ What This Code Does:

  • Line 8: Declares Animator reference
  • Line 14: Gets Animator from child GameObject
  • Line 20: Calls HandleAnimations every frame
  • Line 36: Checks if velocity is not zero
  • Line 39: Updates Animator, triggering transitions
🎯 Testing: Enter Play Mode and move your character. The animation should switch from Idle to Move instantly when you press A/D!
