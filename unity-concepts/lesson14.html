3. The Single Script Problem - When It Breaks Down 🔧

Concept:

Single-script architecture means putting all your player functionality (movement, jumping, attacking, skills) in one PlayerController.cs file. This works for simple games but becomes unmanageable for complex character controllers.

Current Crash Course Architecture

PlayerController.cs (Monolithic Approach)
public class PlayerController : MonoBehaviour
{
    // Movement variables
    public float moveSpeed = 5f;
    public float jumpForce = 10f;
    private Rigidbody2D rb;
    
    // Attack variables
    public GameObject attackHitbox;
    public float attackCooldown = 0.5f;
    private float lastAttackTime;
    private bool isAttacking;
    
    // Animation
    private Animator anim;
    
    void Update()
    {
        // Handle movement (ONLY if not attacking)
        if (!isAttacking)
        {
            float move = Input.GetAxis("Horizontal");
            rb.velocity = new Vector2(move * moveSpeed, rb.velocity.y);
        }
        
        // Handle jump (ONLY if not attacking)
        if (Input.GetKeyDown(KeyCode.Space) && !isAttacking)
        {
            rb.velocity = new Vector2(rb.velocity.x, jumpForce);
        }
        
        // Handle attack
        if (Input.GetKeyDown(KeyCode.Mouse0))
        {
            if (Time.time > lastAttackTime + attackCooldown)
            {
                Attack();
            }
        }
    }
    
    void Attack()
    {
        isAttacking = true;
        rb.velocity = Vector2.zero; // Stop movement
        anim.SetTrigger("Attack");
        Invoke("AttackFinished", 0.4f);
    }
    
    void AttackFinished()
    {
        isAttacking = false;
    }
}

// THIS WORKS... for now. But watch what happens when we add more!

🎯 Current State

  • Works Fine: Our crash course script is perfectly adequate right now
  • Simple Logic: Movement + Jump + Attack = manageable in one script
  • Clear Code: Easy to read and understand for beginners
  • But... This doesn't scale when adding more features
⚠️ Warning: The script above is fine NOW, but it's a ticking time bomb. Once you start adding dash, wall jump, skills, combos, etc., everything falls apart.

4. Simple Games vs Complex Games 🎮

Concept:

When is single-script okay? If your game has simple mechanics that don't interact much, one script is fine. But RPGs with multiple skills, combos, and state-dependent actions need better architecture.

When Single Script Is FINE

Simple Game Examples (Single Script OK)
✅ BASIC PLATFORMER:
- Move left/right
- Jump
- Maybe double jump
→ Total: 2-3 actions, no complex interactions
→ Single script works great!

✅ FLAPPY BIRD CLONE:
- Tap to flap
- Die on collision
→ Total: 2 actions
→ Single script is perfect!

✅ SIMPLE SHOOTER:
- Move
- Shoot
- Reload
→ Total: 3 actions, minimal state management
→ Single script is acceptable!

COMMON PATTERN: Few actions, simple rules
"You can always do X unless Y"

When Single Script BREAKS

Complex Game Examples (Single Script FAILS)
❌ RPG CHARACTER CONTROLLER:
- Walk, Run, Sprint
- Jump, Double Jump, Wall Jump
- Dash, Slide, Roll
- Attack, Heavy Attack, Combo Attack
- Block, Parry, Counter
- Skill 1, Skill 2, Skill 3, Ultimate
- Climbing, Swimming, Flying
→ Total: 20+ actions
→ Complex state rules:
"Can't jump while dashing unless in air"
"Can't attack while wall jumping"
"Can only parry if not in skill animation"
→ Single script becomes NIGHTMARE!

❌ FIGHTING GAME:
- Light/Medium/Heavy attacks
- Special moves with input combinations
- Blocking high/low
- Dodge, Counter, Grab
→ Total: 15+ actions with frame-perfect timing
→ Single script is impossible to manage!

COMMON PATTERN: Many actions, complex interdependencies
"You can do X unless Y or Z, but if W then..."

🎯 The Decision Rule

  • Few Actions (2-5): Single script is fine
  • Medium Actions (5-10): Single script gets messy but manageable
  • Many Actions (10+): Single script becomes unmanageable—use FSM
  • Complex Rules: If actions have many "can't do X while Y" rules, use FSM

5. The Complexity Trap - "Patching a Leaking Barrel" 🛢️

Concept:

The leaking barrel metaphor: When you add features to a single-script architecture, each new feature creates bugs elsewhere. You spend more time preventing breakage than creating features. It's like patching holes in a barrel—fix one, three more appear.

How Complexity Spirals Out of Control

Adding Features to Single Script (The Nightmare)
// ITERATION 1: Simple movement + attack (WORKS FINE)
void Update()
{
    if (!isAttacking)
    {
        Move();
    }
    
    if (Input.GetKeyDown(KeyCode.Mouse0))
    {
        Attack();
    }
}

// ITERATION 2: Add Dash (NOW YOU NEED MORE CHECKS)
void Update()
{
    if (!isAttacking && !isDashing)  // Added isDashing check
    {
        Move();
    }
    
    if (Input.GetKeyDown(KeyCode.Mouse0) && !isDashing)  // Can't attack while dashing
    {
        Attack();
    }
    
    if (Input.GetKeyDown(KeyCode.LeftShift) && !isAttacking)  // Can't dash while attacking
    {
        Dash();
    }
}

// ITERATION 3: Add Wall Jump (COMPLEXITY EXPLODES)
void Update()
{
    if (!isAttacking && !isDashing && !isWallJumping)  // Another condition!
    {
        Move();
    }
    
    if (Input.GetKeyDown(KeyCode.Mouse0) && !isDashing && !isWallJumping)
    {
        Attack();
    }
    
    if (Input.GetKeyDown(KeyCode.LeftShift) && !isAttacking && !isWallJumping)
    {
        Dash();
    }
    
    if (Input.GetKeyDown(KeyCode.Space) && isNearWall && !isDashing && !isAttacking)

        WallJump();
    }

// ITERATION 4: Add Skill System (UNMANAGEABLE)

{

    {
        Move();
    }
    
    // And it goes on... and on... and on...
    // You spend more time adding checks than writing features
}

// Fix one bug → Create three more
// Add one feature → Break two existing features

6. Introduction to Finite State Machine (FSM) 🔄

Concept:

Finite State Machine (FSM) is an architectural pattern where your character can be in exactly ONE state at a time. Instead of checking "not attacking AND not dashing AND not jumping", you just enable the current state and disable all others.

What Is a State?

Understanding States
STATE = A distinct condition or behavior

PLAYER STATES:
- IdleState
- MoveState
- JumpState
- DashState
- AttackState
- WallJumpState
- SkillState

FINITE = Limited number of states
STATE MACHINE = System that manages which state is active

CORE PRINCIPLE:
Player can only be in ONE state at a time

Example:
If player is in "AttackState":
→ MoveState is DISABLED (can't move)
→ JumpState is DISABLED (can't jump)
→ DashState is DISABLED (can't dash)
→ Only AttackState code runs!

When attack finishes:
→ Switch to IdleState or MoveState
→ Now only that state's code runs!

FSM vs Boolean Checks

Comparison
// ❌ BOOLEAN HELL (Single Script)
void Update()
{
    // Move logic
    if (!isAttacking && !isDashing && !isWallJumping && !isCastingSkill)
    {
        Move();
    }
    
    // Attack logic
    if (Input.GetKeyDown(KeyCode.Mouse0) && !isDashing && !isWallJumping)
    {
        Attack();
    }
    
    // Every action needs to check every other action!
}

// ✅ FSM ELEGANCE (State Machine)
void Update()
{
    // Only the current state's Update() runs!
    currentState.Update();
    
    // That's it! No boolean checks needed!
}

// In AttackState.cs
public override void Update()
{
    // Only attack logic here
    // No checks needed because we KNOW we're attacking
    // Other states are automatically disabled!
}

🎯 FSM Benefits

  • One State Active: Only one script runs at a time—no conflicts
  • Clean Code: Each state file contains only that state's logic
  • Easy to Debug: Know exactly which code is running
  • Scalable: Add new states without breaking existing ones
  • Professional: Industry-standard pattern for complex characters

7. FSM Architecture - One Script Per Action 📁

Concept:

FSM separates concerns by giving each state its own script. JumpState.cs only handles jumping, MoveState.cs only handles movement, etc. This makes code modular, testable, and easy to maintain.

File Structure

Project Organization
Scripts/
├── Player/
│ ├── PlayerStateMachine.cs // Manages which state is active
│ ├── PlayerState.cs // Base class all states inherit from
│ └── States/
│ ├── PlayerIdleState.cs // ONLY idle logic
│ ├── PlayerMoveState.cs // ONLY movement logic
│ ├── PlayerJumpState.cs // ONLY jump logic
│ ├── PlayerDashState.cs // ONLY dash logic
│ ├── PlayerAttackState.cs // ONLY attack logic
│ └── PlayerWallJumpState.cs // ONLY wall jump logic

COMPARE TO SINGLE SCRIPT:
Scripts/
├── Player/
│ └── PlayerController.cs // Everything in one file (2000+ lines!)

FSM = 6 small focused files (100-200 lines each)
Single = 1 giant file (1500-2000+ lines)

Example: Jump State

PlayerJumpState.cs (Complete File)
public class PlayerJumpState : PlayerState
{
    public PlayerJumpState(PlayerStateMachine stateMachine) : base(stateMachine) { }
    
    public override void Enter()
    {
        // Runs once when entering jump state
        player.rb.velocity = new Vector2(player.rb.velocity.x, player.jumpForce);
        player.anim.SetBool("isJumping", true);
    }
    
    public override void Update()
    {
        // Runs every frame while in jump state
        
        // Check if we've landed → switch to Idle/Move state
        if (player.IsGrounded())
        {
            stateMachine.ChangeState(player.idleState);
        }
        
        // Allow air control (move left/right while jumping)
        float move = Input.GetAxis("Horizontal");
        player.rb.velocity = new Vector2(move * player.airControlSpeed, player.rb.velocity.y);
    }
    
    public override void Exit()
    {
        // Runs once when leaving jump state
        player.anim.SetBool("isJumping", false);
    }
}

// That's the ENTIRE jump script!
// Simple, focused, easy to understand
// Even if you've never seen this code before, you can tell it makes the player jump!

🎯 State Script Structure

  • Enter(): Runs once when state becomes active (setup)
  • Update(): Runs every frame while state is active (main logic)
  • Exit(): Runs once when leaving state (cleanup)
  • Focused: Each file does ONE thing and does it well
💡 Pro Tip: Even if you've never looked at this code before, you can instantly tell PlayerJumpState.cs makes the player jump. Compare that to a 2000-line PlayerController.cs where you have to hunt for jump logic!

8. Comparing Monolithic vs FSM Code Structure 🔍

Concept:

Side-by-side comparison shows why FSM scales better. Monolithic (single-script) architecture requires exponentially more boolean checks as features grow. FSM stays clean no matter how many states you add.

Adding a New Feature: Wall Jump

❌ Monolithic Approach (Painful)
// BEFORE: PlayerController.cs (500 lines)
// AFTER adding wall jump: PlayerController.cs (700 lines)

public class PlayerController : MonoBehaviour
{
    // New variables
    private bool isWallJumping;
    private float wallJumpTime;
    // ... 10 more wall jump variables
    
    void Update()
    {
        // Modify EVERY existing check
        if (!isAttacking && !isDashing && !isWallJumping)  // ← Added !isWallJumping
        {
            Move();
        }
        
        if (Input.GetKeyDown(KeyCode.Mouse0) && !isDashing && !isWallJumping)  // ← Modified
        {
            Attack();
        }
        
        if (Input.GetKeyDown(KeyCode.LeftShift) && !isAttacking && !isWallJumping)  // ← Modified
        {
            Dash();
        }
        
        // New wall jump logic (50+ lines)
        if (Input.GetKeyDown(KeyCode.Space) && !isAttacking && !isDashing)
        {
            if (CheckWallContact())
            {
                WallJump();
            }
        }
        
        if (isWallJumping)
        {
            // Wall jump movement logic
            // Handle wall jump direction
            // Check for wall jump end condition
            // ... 30 more lines
        }
    }
    
    // New wall jump methods
    void WallJump() { /* ... */ }
    bool CheckWallContact() { /* ... */ }
    void EndWallJump() { /* ... */ }
    // ... 5 more wall jump methods
}

// RESULT:
// - Modified 15 existing lines
// - Added 200+ new lines
// - Introduced 3 new bugs in existing features
// - Spent 4 hours debugging
// - Afraid to touch code now
✅ FSM Approach (Easy)
// BEFORE: 6 state files
// AFTER adding wall jump: 7 state files

// NEW FILE: PlayerWallJumpState.cs (120 lines)
public class PlayerWallJumpState : PlayerState
{
    public PlayerWallJumpState(PlayerStateMachine stateMachine) : base(stateMachine) { }
    
    public override void Enter()
    {
        // Wall jump initialization
        Vector2 wallJumpDirection = player.IsTouchingRightWall() ? Vector2.left : Vector2.right;
        player.rb.velocity = new Vector2(wallJumpDirection.x * player.wallJumpForce, player.wallJumpForce);
        player.anim.SetTrigger("WallJump");
    }
    
    public override void Update()
    {
        // Wall jump logic (only runs when wall jumping!)
        if (player.IsGrounded())
        {
            stateMachine.ChangeState(player.idleState);
        }
        
        // Allow slight air control
        float move = Input.GetAxis("Horizontal");
        player.rb.velocity = new Vector2(
            Mathf.Lerp(player.rb.velocity.x, move * player.airControlSpeed, 0.1f),
            player.rb.velocity.y
        );
    }
    
    public override void Exit()
    {
        // Cleanup
    }
}

// MODIFY: PlayerJumpState.cs (Add 3 lines to check for wall)
public override void Update()
{
    // Existing jump logic...
    
    // NEW: Can transition to wall jump if touching wall
    if (Input.GetKeyDown(KeyCode.Space) && player.IsTouchingWall())
    {
        stateMachine.ChangeState(player.wallJumpState);  // ← Only change!
    }
}

// RESULT:
// - Created 1 new file (120 lines)
// - Modified 1 existing file (added 3 lines)
// - Zero bugs introduced
// - Took 1 hour
// - Other states completely unaffected!

Scalability Comparison

Adding Features Over Time
MONOLITHIC (Single Script):
Feature 1 (Attack): +100 lines, 0 bugs, 1 hour
Feature 2 (Dash): +150 lines, 1 bug, 2 hours
Feature 3 (Wall Jump): +200 lines, 3 bugs, 4 hours
Feature 4 (Skill): +300 lines, 5 bugs, 8 hours
Feature 5 (Combo): +400 lines, 10 bugs, 16 hours

Total: 1150 lines, 19 bugs, 31 hours
Final file: 2000+ lines of spaghetti code

FSM (State Machine):
Feature 1 (Attack): New file 120 lines, 0 bugs, 1 hour
Feature 2 (Dash): New file 100 lines, 0 bugs, 1 hour
Feature 3 (Wall Jump): New file 120 lines, 0 bugs, 1 hour
Feature 4 (Skill): New file 150 lines, 0 bugs, 2 hours
Feature 5 (Combo): New file 180 lines, 0 bugs, 2 hours

Total: 5 separate files (670 lines), 0 bugs, 7 hours

FSM = 4x faster, 0 bugs, maintainable
Monolithic = 4x slower, 19 bugs, unmaintainable

🎯 Why FSM Wins for Complex Games

  • Isolation: New features don't affect existing features
  • Speed: Add features 4x faster after initial setup
  • Quality: Dramatically fewer bugs
  • Maintenance: Easy to find and fix issues
  • Teamwork: Multiple developers can work on different states
