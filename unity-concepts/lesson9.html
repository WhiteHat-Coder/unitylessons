1. Creating the Attack Animation ⚔️

Concept:

An attack animation shows your character performing a combat action. Like other platformer animations, it should be quick and responsive. Most sprite sheets contain multiple attack variations (light, heavy, special), but we'll start with one to learn the core mechanics.

Creating the Animation Clip

Setup Process
1. Select Animator in Hierarchy
2. Animation window → Create New Clip
3. Save in Animations folder as "Player_Attack"
4. Go to Graphics folder
5. Find attack sprites (typically frames 14-20)
6. Select one attack sequence (first attack)
7. Drag sprites to timeline
8. Set Sample Rate: 15

🎯 Selecting Attack Frames

  • Start Frame: Character begins swing/strike
  • Middle Frames: Weapon/fist extends outward
  • Impact Frame: Peak of attack motion
  • Recovery Frames: Character pulls weapon back
  • Avoid Mixing: Don't combine different attack types in one animation

Why Sample Rate 15 for Attacks?

Attack Speed Comparison
Sample Rate 12: Standard speed (used for idle/run)
Sample Rate 15: Slightly faster (good for attacks)
Sample Rate 18+: Very fast (for quick jabs)

Attacks should feel quicker than movement animations
Makes combat feel responsive and impactful

Adding Weapon Return Frames

Polish Technique:

After the attack peak, add a few frames showing the character pulling the weapon back to idle position. This creates a complete attack arc instead of an abrupt cut back to idle.

Recovery Frames
Original: Start → Swing → Impact → END (abrupt)

Improved: Start → Swing → Impact → Pull Back → Ready

Add 2-3 frames after impact:
- Frame showing weapon beginning to return
- Frame showing weapon halfway back
- Frame showing ready stance

Result: Smooth, complete attack motion
🎨 Animation Polish: Recovery frames make attacks feel "complete" rather than choppy. Test with and without them—the difference is immediately noticeable!

2. Understanding Animation Triggers 🎬

Concept:

Animation Triggers are special Animator parameters designed for one-time events. Unlike booleans that stay true/false, triggers automatically reset after being used. Perfect for actions like attacking, taking damage, or opening doors.

Parameter Types Comparison

Four Parameter Types
BOOL (true/false):
- Stays in state until manually changed
- Example: isGrounded, isMoving
- Use for: Continuous states

FLOAT (decimal numbers):
- Continuous value range
- Example: xVelocity, yVelocity
- Use for: Blend trees, speed values

INT (whole numbers):
- Discrete numbered states
- Example: comboStep (1, 2, 3)
- Use for: State machines with numbered stages

TRIGGER (one-time event):
- Fires once, then auto-resets
- Example: Attack, TakeDamage, Jump
- Use for: Momentary actions

✨ Why Triggers Are Perfect for Attacks

  • One-Time Action: Attack happens once per button press
  • Auto-Reset: Don't need to manually set it back to false
  • No Leftover State: Can't get "stuck" in attack mode
  • Clean Code: Just call SetTrigger, no boolean management
  • Spam Protection: Even if called multiple times, triggers once per frame

Creating the Attack Trigger

Animator Setup
1. Open Animator window
2. Go to Parameters tab
3. Click "+" button
4. Select "Trigger"
5. Name it: "Attack"

Now you can use this in transitions!
💡 Naming Consistency: Name your trigger exactly "Attack" (capital A). When you call it from code, the spelling must match perfectly, including capitalization!

3. Setting Up Attack Transitions 🔄

Concept:

Attack animations need two transitions: entering the attack (from Idle_Move) and exiting the attack (back to Idle_Move). The enter transition uses the trigger, while the exit transition uses "Has Exit Time" to play the full animation.

Transition to Attack (Entry)

Entry Transition Setup
1. Right-click "Idle_Move" → Make Transition → "Player_Attack"
2. Click the transition arrow
3. In Inspector:
- Add Condition: Attack (trigger)
- Uncheck "Has Exit Time"
- Set "Transition Duration" to 0

Logic: When Attack trigger fires, instantly switch to attack animation

🎯 Entry Transition Settings Explained

  • Condition = Attack: Only transitions when trigger is fired
  • No Exit Time: Don't wait for current animation to finish
  • Duration = 0: Instant transition for responsive feel
  • Result: Attack starts immediately when player presses button

Transition from Attack (Exit)

Exit Transition Setup
1. Right-click "Player_Attack" → Make Transition → "Idle_Move"
2. Click the transition arrow
3. In Inspector:
- No conditions needed!
- CHECK "Has Exit Time"
- Set "Exit Time" to 1.0
- Set "Transition Duration" to 0

Logic: After animation plays once (exit time = 1), return to idle/move

🎯 Exit Transition Settings Explained

  • Has Exit Time = TRUE: Wait for animation to complete
  • Exit Time = 1.0: Means "wait for 100% of animation"
  • No Conditions: Automatically returns when done
  • Duration = 0: Instant return to idle/move
  • Result: Attack plays fully, then returns to normal movement
🎮 Exit Time Values:
Exit Time 0.5 = Transitions halfway through animation
Exit Time 1.0 = Transitions at animation end (most common for attacks)
Exit Time 1.5 = Would wait for animation to loop 1.5 times

Why This Two-Part System Works

Flow Diagram
Player in Idle_Move
↓
Player presses Attack button
↓
Trigger "Attack" fires
↓
Instantly transition to Player_Attack (no exit time)
↓
Attack animation plays completely (exit time = 1.0)
↓
Automatically return to Idle_Move (no condition needed)
↓
Player back in normal movement state

4. Implementing Attack Input 🖱️

Concept:

Attacks are typically triggered by mouse clicks in PC games. We'll detect left mouse button input and trigger the attack animation. We also introduce the concept of "trying" to perform actions—the attempt might fail if conditions aren't met.

Adding the Input Check

Handle Input Method
private void HandleInput()
{
    xInput = Input.GetAxisRaw("Horizontal");

    if (Input.GetKeyDown(KeyCode.Space))
    {
        TryToJump();
    }

    if (Input.GetKeyDown(KeyCode.Mouse0)) // Left mouse button
    {
        TryToAttack();
    }
}

🖱️ Mouse Button KeyCodes

  • KeyCode.Mouse0: Left mouse button
  • KeyCode.Mouse1: Right mouse button
  • KeyCode.Mouse2: Middle mouse button (scroll click)
  • GetKeyDown: Detects single press (not held)
  • Perfect for Attacks: One click = one attack

The "Try" Method Naming Convention

Best Practice:

Name methods "TryToAttack" or "TryToJump" instead of just "Attack" or "Jump". This signals that the action might not execute—there are conditions that could prevent it. Clear naming makes code more understandable.

Try Method Implementation
private void TryToAttack()
{
    if (isGrounded)
    {
        anim.SetTrigger("Attack");
    }
}

// Renaming existing Jump method for consistency:
private void TryToJump()
{
    if (isGrounded)
    {
        rb.linearVelocity = new Vector2(rb.linearVelocity.x, jumpForce);
    }
}
🔧 Visual Studio Rename Shortcut: Ctrl+R, R (press R twice) to rename a method across the entire project. Ensures all references update automatically!

Why Only Attack While Grounded?

⚠️ Design Decision

  • Animation Interruption: Attacking in air would interrupt jump/fall animations
  • Visual Glitches: Mid-air attack looks buggy without proper setup
  • State Complexity: Air attacks need separate animation states
  • For Now: Ground-only attacks keep the system simple
  • Later: Can add air attacks as an advanced feature

5. Introduction to Animation Events 🎯

Concept:

Animation Events let you call code at specific frames of an animation. Think of it as placing markers in your animation timeline that trigger functions. Perfect for syncing game logic with animation timing—like disabling movement when attack starts.

The Problem: Moving During Attacks

🤔 Why Current System Fails

  • Attack Triggers: Animation starts playing
  • But Movement Still Active: HandleMovement still applies velocity
  • Result: Character slides around while attacking
  • Looks Terrible: Breaks immersion and game feel
  • Need Solution: Disable movement when attack starts, re-enable when it ends

The Solution: Register Start and End Moments

Animation Event Strategy
CONCEPT:
- At frame 0 (attack starts): Call DisableMovement()
- At last frame (attack ends): Call EnableMovement()

IMPLEMENTATION:
1. Animation Events mark these exact frames
2. Events call functions at those moments
3. Functions control canMove and canJump booleans
4. Movement system respects these flags

Result: Character locked in place during attack

How Animation Events Work

🎬 Animation Event System

  • Timeline Markers: Place events at any frame in animation
  • Function Calls: Events call methods from scripts on same GameObject
  • Perfect Sync: Code executes exactly when animation reaches that frame
  • Multiple Events: Can have many events in one animation
  • Common Uses: Footstep sounds, attack damage hitboxes, particle effects
💡 Key Requirement: Animation Events can ONLY call functions from scripts attached to the same GameObject as the Animator component. This is why we need a helper script!

Why We Need a Separate Script

Architecture Problem:

The Animator is on the Player GameObject, but we can't put all animation event functions in the PlayerController—it would get messy. Better to create a dedicated PlayerAnimationEvents script for clean organization.

Component Structure
Player GameObject:
├─ Animator (controls animations)
├─ PlayerController (movement, jumping, attack logic)
└─ PlayerAnimationEvents (handles animation event callbacks)

Animation Events call methods in PlayerAnimationEvents
PlayerAnimationEvents communicates with PlayerController
PlayerController enables/disables movement

7. Boolean Flags for Movement Control 🚦

Concept:

Boolean flags act as permission gates in your code. canMove and canJump control whether the player is allowed to move or jump. Various game systems can toggle these flags to temporarily disable player control.

Adding the Boolean Variables

PlayerController Variables
// Add these near the top of PlayerController
private bool canMove = true;
private bool canJump = true;

// Start as true because player should be able to move from game start

🎯 Use Cases for These Flags

  • During Attacks: Set canMove = false to prevent sliding
  • Taking Damage: Disable both during hit stun animation
  • Cutscenes: Disable player control during story moments
  • Dialogue: Stop movement while talking to NPCs
  • Death State: Disable everything when character dies

Modifying HandleMovement

Adding Movement Gate
private void HandleMovement()
{
    if (canMove == true) // Can also write: if (canMove)
    {
        rb.linearVelocity = new Vector2(xInput * moveSpeed, rb.linearVelocity.y);
    }
}
🎓 Boolean Shorthand:
if (canMove == true) ← Explicit, easier for beginners
if (canMove) ← Shorthand, same meaning
if (canMove == false) ← Explicit false check
if (!canMove) ← Shorthand, "if NOT canMove"

Modifying TryToJump

Adding Jump Gate
private void TryToJump()
{
    if (isGrounded && canJump) // Both conditions must be true
    {
        rb.linearVelocity = new Vector2(rb.linearVelocity.x, jumpForce);
    }
}

📊 The && (AND) Operator Review

  • Both Must Be True: isGrounded AND canJump must both be true
  • If Either False: Jump doesn't execute
  • Safety Check: Multiple conditions prevent unwanted jumps
  • Example: On ground but attacking = isGrounded true, canJump false = no jump

The Problem: Existing Velocity

Critical Issue:

If the player is moving when they attack, canMove = false prevents NEW input, but the Rigidbody still has velocity from BEFORE the attack. Character continues sliding! We need to manually stop the velocity.

Stopping Velocity on Attack
private void TryToAttack()
{
    if (isGrounded)
    {
        anim.SetTrigger("Attack");
        
        // CRITICAL: Stop horizontal velocity immediately
        rb.linearVelocity = new Vector2(0, rb.linearVelocity.y);
    }
}
⚠️ Common Mistake: Forgetting to reset velocity causes "attack sliding." Always zero out velocity when starting actions that should stop movement!

8. Public Methods for State Control 🎛️

Concept:

We need a way for OTHER scripts (like PlayerAnimationEvents) to control the player's movement state. Public methods provide a clean interface for enabling/disabling movement and jumping from outside the PlayerController.

Creating the Control Method

Public Enable/Disable Method
// In PlayerController script

public void EnableMovementAndJump(bool enable)
{
    canMove = enable;
    canJump = enable;
}

// Called with true: Both become true (enabled)
// Called with false: Both become false (disabled)

🎯 How This Works

  • Parameter 'enable': Passed value (true/false) controls both flags
  • Call with true: Re-enables movement and jumping
  • Call with false: Disables both actions
  • Single Method: Controls both booleans together (usually want both)
  • Public Access: Any script can call this method

Why Public Instead of Private?

Access Level Comparison
PRIVATE METHOD:
- Only callable from within the same script
- Internal implementation detail
- Example: HandleMovement(), HandleInput()

PUBLIC METHOD:
- Callable from ANY script
- Interface/API for other systems
- Example: EnableMovementAndJump(), TakeDamage()

Our case: PlayerAnimationEvents needs to call this
Must be public!
🎓 Encapsulation Principle: Make methods private by default. Only make them public when other scripts need access. Keeps code organized and prevents unintended calls.

Using Parameters for Flexibility

Parameter-Based Design
// ONE METHOD instead of TWO:
public void EnableMovementAndJump(bool enable) { ... }

// Instead of separate methods:
// public void EnableMovement() { ... }
// public void DisableMovement() { ... }

Called from other scripts:
playerController.EnableMovementAndJump(false); // Disable
playerController.EnableMovementAndJump(true); // Enable

✨ Benefits of Parameter Approach

  • Less Code: One method instead of two
  • Clear Intent: Caller explicitly passes true/false
  • Flexible: Value can come from variable, not hardcoded
  • Maintainable: One place to modify the logic

9. GetComponentInParent Explained 👨‍👦

Concept:

GetComponentInParent searches UP the hierarchy tree to find a component on a parent GameObject. Essential when your script is on a child object but needs to communicate with components on the parent.

Understanding GameObject Hierarchy

Typical Character Setup
Player (Parent GameObject)
├─ PlayerController script
├─ Rigidbody2D
├─ Capsule Collider 2D
├─ Animator
└─ PlayerAnimationEvents script

PlayerVisuals (Child GameObject)
└─ Sprite Renderer

Common in 2D: Logic on parent, visuals on child

🔍 GetComponent Variants

  • GetComponent: Searches SAME GameObject only
  • GetComponentInChildren: Searches child GameObjects
  • GetComponentInParent: Searches parent GameObjects
  • All return: First matching component found
  • All require: Component type in angle brackets

Implementing in PlayerAnimationEvents

Getting Reference to PlayerController
using UnityEngine;

public class PlayerAnimationEvents : MonoBehaviour
{
    private PlayerController player;

    private void Awake()
    {
        player = GetComponentInParent<PlayerController>();
    }
}
⚠️ Important: Since PlayerAnimationEvents is on the SAME GameObject as PlayerController in our setup, we could use GetComponent. But if you ever move PlayerAnimationEvents to a child, GetComponentInParent still works!

Why Use Awake for References?

⏱️ Unity Lifecycle: Awake vs Start

  • Awake: Called FIRST, before any Start methods
  • Best for: Getting references, initializing variables
  • Guaranteed: Always runs before object is used
  • Start: Called after all Awake methods complete
  • Best for: Setup that depends on other objects' Awake
  • Our case: Awake perfect for GetComponent calls

What If Component Not Found?

Error Handling
private void Awake()
{
    player = GetComponentInParent<PlayerController>();
    
    // Optional: Check if found
    if (player == null)
    {
        Debug.LogError("PlayerController not found on parent!");
    }
}

10. Implementing Disable and Enable Methods 🎬

Concept:

Now we create wrapper methods in PlayerAnimationEvents that will be called by animation events. These methods simply forward the call to PlayerController's EnableMovementAndJump with appropriate true/false values.

The Complete PlayerAnimationEvents Script

Full Implementation
using UnityEngine;

public class PlayerAnimationEvents : MonoBehaviour
{
    private PlayerController player;

    private void Awake()
    {
        player = GetComponentInParent<PlayerController>();
    }

    // Called by animation event at start of attack
    private void DisableMovementAndJump()
    {
        player.EnableMovementAndJump(false);
    }

    // Called by animation event at end of attack
    private void EnableMovementAndJump()
    {
        player.EnableMovementAndJump(true);
    }
}

🎯 Method Flow

  • Animation Event: Triggers at specific frame
  • Calls: DisableMovementAndJump() in PlayerAnimationEvents
  • Which Calls: player.EnableMovementAndJump(false)
  • Which Sets: canMove = false, canJump = false
  • Result: Player can't move or jump

Lambda Expression Alternative

Shorthand Syntax
// Traditional method syntax:
private void DisableMovementAndJump()
{
    player.EnableMovementAndJump(false);
}

// Lambda expression (one-liner):
private void DisableMovementAndJump() => player.EnableMovementAndJump(false);

// Both do exactly the same thing!
📚 Lambda Expressions: When a method contains only ONE line of code, you can use the => (arrow) syntax. Good for keeping scripts compact. The => means "this method returns/executes this expression."

✨ When to Use Lambda Expressions

  • Single Line: Only works for one-line methods
  • Space Saving: Reduces vertical space in long scripts
  • Readability: Cleaner for simple wrapper/delegation methods
  • Personal Choice: Both styles are correct, pick what you prefer
  • Good Practice: Start developing the habit now for professional code

12. Final Polish: Preventing Slide with Else 🎨

Concept:

Even with canMove = false, physics materials can cause residual sliding. The solution: add an else statement to HandleMovement that actively sets X velocity to zero when movement is disabled.

The Slippery Material Problem

⚠️ Physics Interaction Issue

  • We Have: Slippery Physics Material (friction = 0) from Lesson 8
  • Problem: Zero friction means character continues sliding slightly
  • During Attack: canMove = false stops NEW input, not existing velocity
  • Result: Small "drift" during attack animation
  • Solution: Actively force velocity to zero

Adding the Else Statement

Complete HandleMovement
private void HandleMovement()
{
    if (canMove)
    {
        // Allow movement
        rb.linearVelocity = new Vector2(xInput * moveSpeed, rb.linearVelocity.y);
    }
    else
    {
        // Force stop movement
        rb.linearVelocity = new Vector2(0, rb.linearVelocity.y);
    }
}

🎯 Why This Works

  • If canMove = true: Normal movement with xInput
  • If canMove = false: X velocity forced to 0 every frame
  • Continuous Enforcement: Every Update() frame, velocity reset
  • No Drift: Character completely locked in place
  • Y Velocity Preserved: Gravity still works (falling, etc.)
💡 Design Pattern: If/else for "enabled" vs "disabled" states is common. Enabled does normal logic, disabled enforces locked state. Use this pattern for many game systems!
