1. The Philosophy of 2D Jump Animations 🎮

Concept:

If you've never set up a 2D character before, you might imagine elaborate jump sequences with preparation frames, takeoff, mid-air, transition, landing, and hero landing poses. However, professional 2D platformers use surprisingly simple jump animations—often just 1-3 frames. This is intentional design for responsiveness.

Why Simple Jump Animations Work

🎯 The Responsiveness Principle

  • Instant Feedback: When players press jump, they expect immediate upward motion
  • No Wind-Up: Preparation animations feel sluggish and unresponsive
  • Classic Examples: Mario (1-2 frames), Celeste, Hollow Knight—all use minimal jump frames
  • Player Expectations: Button press = instant action, not a preparation sequence

Historical Context: Super Mario Bros

Mario's Jump Animation
Frame 1: Standing
Frame 2: Arms up (jump pose)
Result: Character immediately in jump state

Total frames: 1-2 frames for entire jump
Feel: Snappy, responsive, arcade-perfect
🎨 Design Lesson: More animation frames ≠ Better feel. In platformers, responsiveness beats realism. Players need to feel their input instantly reflected on screen.

📊 Modern 2D Platformer Standards

  • Jump Animation: 2-4 frames (rising pose)
  • Fall Animation: 2-4 frames (falling pose)
  • No Landing Animation: Immediate transition to idle/move
  • Why It Works: Frames change so fast (60 FPS gameplay) that transitions aren't noticeable

2. Creating the Jump Animation 🚀

Concept:

The jump animation represents your character rising upward. It should capture the feeling of ascent—arms up, body extended, looking upward. We'll select specific frames from our sprite sheet that convey this upward motion.

Creating the Animation Clip

Step-by-Step Setup
1. Select Animator in Hierarchy
2. Animation window → Create New Clip
3. Name: "Player_Jump"
4. Go to Graphics folder in Project window
5. Find jump sprites (frames 41-44 typically)
6. Select first 3 frames (NOT the falling frame)
7. Drag sprites to Player_Jump timeline
8. Set Sample Rate: 12

🎯 Selecting the Right Frames

  • Frame 1: Character starting to rise (arms beginning to lift)
  • Frame 2: Mid-jump (arms fully extended upward)
  • Frame 3: Peak of jump (fully extended pose)
  • EXCLUDE: Any frame where character looks downward (that's fall animation)

Understanding Sample Rate

Sample Rate = Frames Per Second
Sample Rate: 12
= 12 frames per second
= Each sprite shows for 1/12 of a second

3 sprites × (1/12 sec) = 0.25 seconds total

Lower sample rate = Slower animation
Higher sample rate = Faster animation
⚙️ Why 12 FPS? Sample rate of 12 is common for sprite animations because it's fast enough to feel responsive but slow enough to see each frame clearly. Game runs at 60 FPS, but animations can be 12 FPS.

Previewing Your Animation

✅ Quality Check

  • Click Play button in Animation window
  • Character should smoothly transition through rising poses
  • No downward-looking frames should appear
  • Animation should feel quick and punchy

3. Creating the Fall Animation 🪂

Concept:

The fall animation shows your character descending. It captures the opposite feeling from jump—looking down, body preparing to land. This animation loops while the character is falling.

Initial Setup

Creating Fall Animation
1. Animation window → Create New Clip
2. Name: "Player_Fall"
3. Find falling sprites (typically after jump sprites)
4. Select 3-4 falling frames
5. Drag to Player_Fall timeline
6. Set Sample Rate: 12

Dealing with Transition Frames

🤔 The Transition Frame Problem

  • Issue: Some sprite sheets include transition frames (jump → fall)
  • In Slow Motion: Transitions look smooth and natural
  • In Actual Gameplay: Animation is so fast transitions become glitchy
  • Solution: Remove transition frames, keep only clear falling poses
Refined Fall Animation
Original: 4 frames (includes transition)
Problem: Transition frame makes it glitchy at game speed

Refined: 3 frames (pure falling poses)
Result: Clean, crisp falling animation

Keep frames that clearly show:
- Character looking down
- Arms in falling position
- Body anticipating landing
🎨 Pro Tip: When in doubt, test animations at game speed, not in slow preview. What looks good at 1x speed rarely looks good at 60 FPS gameplay speeds.

Loop Time Setting

🔄 Should Fall Animation Loop?

  • Initially: Enable loop time for continuous falling
  • With Blend Trees: Disable loop (we'll control it with velocity)
  • Final Decision: Test both options and choose what feels better
  • Trade-off: Loop = continuous motion, No Loop = static fall pose

4. Introduction to Blend Trees 🌳

Concept:

A Blend Tree is a special Animator state that smoothly blends between multiple animations based on a parameter value. Instead of hard transitions between animations, blend trees create fluid interpolation. This is perfect for jump/fall because we can blend based on vertical velocity.

Blend Trees vs Traditional Transitions

Comparison
TRADITIONAL TRANSITIONS:
- Animation A → Transition → Animation B
- Requires exit time, transition duration
- Hard switch between states
- More complex to manage

BLEND TREE:
- Automatically blends based on parameter
- Smooth interpolation between animations
- Single parameter controls everything
- Much cleaner to manage

✨ Why Use Blend Trees?

  • Smooth Transitions: No hard cuts between animations
  • Parameter-Driven: One float value controls everything
  • Less Arrows: Reduces Animator complexity dramatically
  • Real-Time Blending: Responds instantly to changing values
  • Professional Workflow: Industry standard for complex character animation
💡 Mental Model: Think of a blend tree like a slider. At -1 = Animation A, at +1 = Animation B, at 0 = blend of both. The parameter value determines where on the slider you are.

5. Creating Jump/Fall Blend Tree 🎯

Concept:

We'll create a blend tree that automatically switches between jump and fall animations based on the character's Y velocity. When velocity is positive (moving up), show jump. When velocity is negative (moving down), show fall. Unity handles the blending automatically.

Creating the Blend Tree

Setup Steps
1. In Animator window → Right-click → Create State → From New Blend Tree
2. Name it: "Jump_Fall"
3. Double-click to enter Blend Tree view
4. In Inspector → Click "+" next to Motion field
5. Add Motion (do this twice)
6. First Motion: Player_Jump
7. Second Motion: Player_Fall

Configuring the Parameter

Y Velocity Parameter
1. In Parameters tab → Create new Float parameter
2. Name: "yVelocity"
3. In Blend Tree Inspector → Change "Blend Parameter" to "yVelocity"
4. Uncheck "Automate Thresholds"
5. Set Jump threshold: 0
6. Set Fall threshold: -1

🎯 Understanding Thresholds

  • Threshold 0 (Jump): When yVelocity ≥ 0, play jump animation
  • Threshold -1 (Fall): When yVelocity < 0, blend toward fall animation
  • Automatic Blending: Unity interpolates between animations based on exact value
  • Positive Velocity: Character rising = Jump animation
  • Negative Velocity: Character falling = Fall animation

Why Y Velocity Is Perfect

Velocity Logic
Y Velocity > 0 → Character moving UP → Show Jump
Y Velocity = 0 → Character at peak → Transition point
Y Velocity < 0 → Character moving DOWN → Show Fall

Rigidbody2D.linearVelocity.y provides this automatically!
No manual tracking needed!
🎮 Pro Technique: Using physics values (like velocity) to drive animations creates realistic, responsive character movement. The animation system reacts to actual game physics in real-time!

Testing the Blend Tree

✅ Visual Testing

  • In Blend Tree view, drag the yVelocity slider in Inspector
  • Positive values → See jump animation
  • Negative values → See fall animation
  • Watch the smooth transition between them

6. Understanding Velocity-Based Animation Control ⚡

Concept:

Velocity-based animation means using the character's actual movement speed (from the physics system) to control which animation plays. This creates perfectly synchronized animation and movement—no lag, no manual tracking, just pure physics-driven animation.

The Power of Direct Physics Integration

Traditional vs Velocity-Based
TRADITIONAL METHOD:
if (jumping) { PlayAnimation("Jump"); }
if (falling) { PlayAnimation("Fall"); }
Problem: Manual state tracking, can desync from physics

VELOCITY-BASED METHOD:
animator.SetFloat("yVelocity", rb.linearVelocity.y);
Benefit: Animation always matches actual physics state

✨ Advantages

  • Always Accurate: Animation reflects true movement state
  • No Manual State: Don't need isJumping, isFalling booleans
  • Handles Edge Cases: Works for knockback, bouncing, any vertical movement
  • Less Code: One line updates the entire system
  • Responsive: Instant reaction to physics changes

How Rigidbody2D.linearVelocity Works

Velocity Vector
Rigidbody2D.linearVelocity → Vector2(x, y)

linearVelocity.x → Horizontal speed
- Positive = Moving right
- Negative = Moving left
- Zero = Stationary horizontally

linearVelocity.y → Vertical speed
- Positive = Moving up (jumping)
- Negative = Moving down (falling)
- Zero = Peak of jump or on ground
🎓 Physics Insight: Velocity is automatically managed by Unity's physics engine. Gravity decreases Y velocity over time, making it naturally transition from positive (jump) to negative (fall). Perfect for animation!

7. Creating Idle/Move Blend Tree with X Velocity 🏃

Concept:

Now that we understand blend trees, let's rebuild our idle/move system using a blend tree with X velocity. This is much cleaner than the transition-based approach from earlier lessons, and it uses a brilliant trick: the same move animation for both left and right movement.

Why Rebuild with Blend Trees?

🎯 The Problem with Traditional Transitions

  • Arrow Spaghetti: Idle → Move, Move → Idle, both → Jump/Fall = 6+ arrows
  • Hard to Manage: Each transition needs conditions, exit time, duration
  • Not Scalable: Adding new states multiplies complexity
  • Learning Value: You needed to understand transitions first, now we optimize

Creating the Blend Tree

Setup Process
1. Delete old Idle and Move states
2. Create State → From New Blend Tree
3. Name: "Idle_Move"
4. Double-click to enter
5. Create new Float parameter: "xVelocity"
6. Set Blend Parameter to "xVelocity"
7. Add Motion field THREE times
8. Uncheck "Automate Thresholds"

The Three-Point Configuration

Three Motion Setup
Motion 1: Player_Move → Threshold: -1
Motion 2: Player_Idle → Threshold: 0
Motion 3: Player_Move → Threshold: 1

Why same animation at -1 and 1?
Because character flipping is handled by FlipX!
Moving left (-1) and moving right (+1) use SAME animation!

🎨 How It Works

  • xVelocity = -1 (or less): Moving left → Play move animation
  • xVelocity = 0: Stationary → Play idle animation
  • xVelocity = 1 (or more): Moving right → Play move animation
  • In Between Values: Blend smoothly between animations

Testing the Blend Tree

Visual Testing
In Blend Tree view, drag xVelocity slider:

Slider at -1 → Character running (will face left via FlipX)
Slider at 0 → Character idle
Slider at +1 → Character running (will face right via FlipX)

Smooth transitions throughout!
💡 Brilliant Design: This three-point setup means we only need ONE move animation instead of separate left/right animations. Character direction is handled by FlipX (from Lesson 6), not separate animations!

8. Setting Up Grounded/Airborne Transitions 🔄

Concept:

Now we have two blend trees: Idle_Move (grounded) and Jump_Fall (airborne). We need transitions between them based on one simple condition: Is the character on the ground? We already have isGrounded from Lesson 7!

Setting Default State

Initial Configuration
1. Right-click "Idle_Move" → Set as Layer Default State
2. Idle_Move turns orange (default state)
3. Character starts in this state when game begins

Transition to Airborne (Jump/Fall)

Ground → Air Transition
1. Right-click "Idle_Move" → Make Transition → "Jump_Fall"
2. Click the transition arrow
3. In Inspector, add Condition:
- isGrounded = false
4. Uncheck "Has Exit Time"
5. Set "Transition Duration" to 0

Logic: When character leaves ground, instantly switch to Jump_Fall

🎯 Why These Settings?

  • Has Exit Time = OFF: Don't wait for animation to finish, transition immediately
  • Transition Duration = 0: Instant switch, no blending (we want snappy response)
  • Condition isGrounded = false: Only transition when leaving ground
  • Result: Instant, responsive transition when jumping or falling off ledge

Transition to Grounded (Idle/Move)

Air → Ground Transition
1. Right-click "Jump_Fall" → Make Transition → "Idle_Move"
2. Click the transition arrow
3. In Inspector, add Condition:
- isGrounded = true
4. Uncheck "Has Exit Time"
5. Set "Transition Duration" to 0

Logic: When character touches ground, instantly switch to Idle_Move
🎮 Design Philosophy: Zero transition duration on grounded/airborne switches creates that tight, responsive platformer feel. Players expect instant ground detection feedback!

The Complete State Machine

Final Animator Structure
Idle_Move (Blend Tree - xVelocity)
├─ Player_Idle (threshold: 0)
├─ Player_Move (threshold: -1)
└─ Player_Move (threshold: 1)

Jump_Fall (Blend Tree - yVelocity)
├─ Player_Jump (threshold: 0)
└─ Player_Fall (threshold: -1)

Transitions:
Idle_Move → Jump_Fall: isGrounded = false
Jump_Fall → Idle_Move: isGrounded = true

✨ Why This Is Elegant

  • Only 2 States: Down from 4 separate animation states
  • Only 2 Transitions: Instead of 8+ transition arrows
  • Automatic Blending: Velocity handles all internal animation switching
  • One Condition: isGrounded controls the entire system

9. Clean Animation Control Code 💻

Concept:

Now we update our PlayerController script to feed velocity data to the Animator. This is where everything comes together—physics drives animation, and we only need three lines of code to control the entire animation system!

Updating HandleAnimation Method

Complete Animation Code
private void HandleAnimation()
{
    anim.SetBool("isGrounded", isGrounded);
    anim.SetFloat("xVelocity", rb.linearVelocity.x);
    anim.SetFloat("yVelocity", rb.linearVelocity.y);
}

🎯 What Each Line Does

  • Line 1 - isGrounded: Controls transition between Idle_Move and Jump_Fall states
  • Line 2 - xVelocity: Controls Idle_Move blend tree (idle vs moving, left vs right)
  • Line 3 - yVelocity: Controls Jump_Fall blend tree (jumping vs falling)
  • Total Control: Three parameters control 4+ animations seamlessly

Removing Old Code

Code Cleanup
// DELETE THESE - No longer needed:
// anim.SetBool("isMoving", ...);
// Manual animation state tracking
// Complex transition logic

// KEEP THESE:
anim.SetBool("isGrounded", isGrounded);
anim.SetFloat("xVelocity", rb.linearVelocity.x);
anim.SetFloat("yVelocity", rb.linearVelocity.y);
✨ Code Beauty: This is incredibly clean! Three lines replace dozens of lines of manual animation control. This is why experienced developers love blend trees—maximum effect, minimum code.

Understanding SetFloat vs SetBool

📊 Animator Parameter Types

  • SetBool: On/off states (isGrounded: true or false)
  • SetFloat: Continuous values (velocity can be any number)
  • SetInt: Whole numbers (useful for state machines with numbered states)
  • SetTrigger: One-time events (like taking damage, not used here)

Complete Player Script Structure

Organized Methods
void Update()
{
    HandleCollision(); // Check ground state
    HandleInput(); // Get player input
    HandleMovement(); // Apply movement
    HandleAnimation(); // Update animation parameters
}

Clean, organized, easy to understand!

Testing at Gameplay Speed

  • Fall Animation: Should clearly show descending

Removing Glitchy Transition Frames

Frame Optimization
PROBLEM DISCOVERED:
Fall animation with 4 frames looks glitchy at game speed
The "transition" frame between jump and fall causes flickering

SOLUTION:
1. Open Player_Fall animation
2. Remove the first 1-2 transition frames
3. Keep only clear "falling" pose frames (3 frames)
4. Result: Clean, crisp animation at gameplay speed
⚡ Performance Tip: Fewer frames = less glitchy = better performance. In fast-paced games, clarity beats complexity. Keep only the frames that clearly communicate the action.

Loop Time Decision

Loop Time Settings
Player_Jump Animation:
Loop Time: OFF (plays once, holds last frame)
Why: Jump is momentary, doesn't need to cycle

Player_Fall Animation:

11. Physics Material 2D: Fixing Wall Sticking 🧱

Concept:

You may notice that when the character jumps against a wall, they stick or slide slowly instead of sliding down smoothly. This is caused by friction between the character's collider and the wall. We fix this with a Physics Material 2D that has zero friction.

Understanding the Problem

🤔 Why Characters Stick to Walls

  • Default Friction: Unity colliders have friction by default
  • Realistic Physics: Friction prevents objects from sliding freely
  • Platformer Problem: We want smooth sliding, not sticking
  • Player Experience: Sticking feels buggy and unresponsive

Creating a Slippery Physics Material

Step-by-Step Creation
1. In Project window → Create folder "Materials"
2. Right-click in Materials folder
3. Create → 2D → Physics Material 2D
4. Name it: "Slippery_Material"
5. Select the material
6. In Inspector, set Friction: 0
7. Leave Bounciness: 0 (we don't want bouncing)

Applying to Character

Application Options
OPTION 1 - Apply to Rigidbody2D:
1. Select Player GameObject
2. Find Rigidbody2D component
3. Drag Slippery_Material to "Material" field
Result: Character has no friction with anything

OPTION 2 - Apply to Collider2D:
1. Select Player GameObject
2. Find Capsule Collider 2D component
3. Drag Slippery_Material to "Material" field
Result: Same effect, different component

RECOMMENDED: Apply to Rigidbody2D (more common practice)

🎯 What Changes

  • Before: Character sticks to walls, slides slowly
  • After: Character slides smoothly down walls
  • Ground: No effect on ground movement (controlled by script)
  • Overall Feel: More responsive, "tighter" controls

Testing the Fix

Wall Slide Test
1. Create a tall wall (duplicate platform, rotate, scale)
2. Enter Play Mode
3. Jump into the wall
4. BEFORE: Character sticks/slides slowly
5. Apply Slippery_Material
6. AFTER: Character slides smoothly down
7. Success! No more wall sticking!
⚙️ Physics Material Properties:
Friction: 0 = slippery, 1 = sticky
Bounciness: 0 = no bounce, 1 = super bouncy
For most platformers: Friction = 0, Bounciness = 0

When NOT to Use Zero Friction

⚠️ Design Considerations

  • Wall Jump Games: Might want slight friction to "catch" walls
  • Ice Levels: Zero friction for slippery floors, high friction for normal floors
  • Climbing Mechanics: Variable friction based on whether climbing
  • Our Case: Simple platformer = zero friction is perfect
