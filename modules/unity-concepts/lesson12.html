Master customizing inherited methods with virtual, override, and base keywords

🎯 What You'll Learn

  • ✅ Understand Polymorphism—the second pillar of OOP
  • ✅ Use the virtual keyword to make methods overridable
  • ✅ Override parent methods with custom child behavior
  • ✅ Call parent implementation with base.Method()

1. What is Polymorphism? 🎭

Concept:

Polymorphism allows you to change the functionality of inherited methods. Two different objects can use the same method name, but what happens inside that method can be completely different. It's the second pillar of Object-Oriented Programming and means "many forms" in Greek.

The Perfect Real-World Analogy

Dancing Example
SCENARIO:
You and I are standing in a room.
Someone walks in and says: "DANCE!"

RESULT:
- You dance in YOUR own way
- I dance in MY own way

OBSERVATION:
✅ Same command: "Dance"
✅ Same action: Dancing
❗ Different execution: Unique styles

This is polymorphism:
ONE COMMAND, DIFFERENT BEHAVIORS

🎯 Polymorphism in Programming

  • Same Method Name: Attack(), Move(), TakeDamage()
  • Different Implementations: Each class does it differently
  • Skeleton.Attack(): Swings sword
  • Goblin.Attack(): Swings sword + steals gold
  • Archer.Attack(): Shoots arrow (completely different)

Why Polymorphism Matters

Without Polymorphism
// ❌ BAD: Need different method names for each enemy
skeleton.MeleeAttack();
goblin.MeleeAttackAndSteal();
archer.ShootArrow();
mage.CastFireball();

PROBLEMS:
- Hard to remember different method names
- Can't treat all enemies the same way
- Loops become complicated
- Not scalable
✅ With Polymorphism
// ✅ GOOD: Same method name, different behaviors
skeleton.Attack(); // Melee
goblin.Attack(); // Melee + Steal
archer.Attack(); // Shoot Arrow
mage.Attack(); // Cast Fireball

// Can loop through all enemies uniformly
foreach (Enemy enemy in allEnemies)
{
    enemy.Attack(); // Each does their version
}

BENEFITS:
✅ Consistent interface
✅ Easy to loop through collections
✅ Clean, readable code
✅ Easy to add new enemy types
🎓 Core Principle: Polymorphism enables "one interface, many implementations." You call the same method on different objects, and each object responds in its own unique way!

2. The Wrong Approach: Creating New Methods ❌

Concept:

Before learning polymorphism, let's see the common mistake beginners make: creating new methods instead of overriding existing ones. This approach works but is inefficient and not professional.

The Problem Scenario

Goblin Should Steal When Attacking
// Goal: When Goblin attacks, also steal gold

// Enemy.cs (base class)
public class Enemy : MonoBehaviour
{
    private void Attack() // ❌ Private - can't access in child
    {
        Debug.Log(enemyName + " attacks!");
    }
}

// EnemyGoblin.cs
public class EnemyGoblin : Enemy
{
    private void StealMoney()
    {
        Debug.Log(enemyName + " steals gold!");
    }

    // ❌ Problem: Attack() is private, can't access it!
}

Beginner Solution: Make Attack Protected

First Attempt
// Enemy.cs
public class Enemy : MonoBehaviour
{
    protected void Attack() // Changed to protected
    {
        Debug.Log(enemyName + " attacks!");
    }
}

// EnemyGoblin.cs
public class EnemyGoblin : Enemy
{
    private void StealMoney()
    {
        Debug.Log(enemyName + " steals gold!");
    }

    // Create NEW method that calls both
    private void SpecialAttack()
    {
        StealMoney();
        Attack(); // ✅ Now accessible because protected
    }

    [ContextMenu("Special Attack")]
    private void SpecialAttack_Test()
    {
        SpecialAttack();
    }
}

⚠️ Why This Works But Is Wrong

  • ✅ Works: You can call SpecialAttack() to do both actions
  • ❌ Two Methods: Now have Attack() AND SpecialAttack()
  • ❌ Inconsistent: Skeleton uses Attack(), Goblin uses SpecialAttack()
  • ❌ Breaks Polymorphism: Can't treat all enemies the same
  • ❌ Not Scalable: Need UltimateAttack(), SuperAttack(), etc. for variations

Testing the Wrong Approach

Results
// Press F key (calls Attack() on all enemies)
Skeleton attacks!
Goblin attacks! // ❌ Doesn't steal - wrong method called

// Right-click Goblin → SpecialAttack
Goblin steals gold!
Goblin attacks! // ✅ Works, but need to manually call SpecialAttack

PROBLEM:
- When Update() calls Attack(), Goblin doesn't steal
- Need to remember to call SpecialAttack() instead
- Code becomes confusing
💡 The Right Way: Instead of creating new methods, we should override the existing Attack() method and customize its behavior for Goblin. That's polymorphism!

3. The Virtual Keyword: Making Methods Overridable 🔓

Concept:

The virtual keyword marks a method as overridable by child classes. By default, methods in parent classes are "sealed"—child classes can't change them. Adding virtual opens them for customization.

Making Attack() Virtual

Enemy.cs - Base Class
using UnityEngine;

public class Enemy : MonoBehaviour
{
    [SerializeField] protected string enemyName = "Enemy";

    // Virtual method - child classes CAN override this
    protected virtual void Attack()
    {
        Debug.Log(enemyName + " attacks!");
    }

    private void Update()
    {
        if (Input.GetKeyDown(KeyCode.F))
        {
            Attack();
        }
    }
}

🔑 Virtual Keyword Effects

  • protected virtual: Method is accessible to children AND overridable
  • Default Behavior: Base implementation still exists
  • Optional Override: Child classes CAN override, but don't have to
  • Not Sealed: Removes the "sealed" status from the method
  • Professional Standard: Use virtual when you expect child classes to customize behavior

What Can Be Virtual?

Virtual Options
public class Enemy : MonoBehaviour
{
    // ✅ Virtual method
    protected virtual void Attack() { }

    // ✅ Virtual property
    public virtual int MaxHealth => 100;

    // ✅ Virtual event (advanced)
    protected virtual void OnDeath() { }

    // ❌ Can't be virtual: static methods
    // public static virtual void Test() { } // ERROR

    // ❌ Can't be virtual: constructors
    // public virtual Enemy() { } // ERROR
}

RULE: Methods, properties, and events can be virtual.
Constructors and static members cannot.
🎨 Design Pattern: Mark methods as virtual when you expect different enemy types to have different implementations. Common virtual methods: Attack(), Die(), TakeDamage(), Move().

Non-Virtual vs Virtual

Comparison
// WITHOUT virtual
protected void Attack() { } // Sealed - can't override

// EnemyGoblin.cs
// ❌ Can't override - method is sealed


// WITH virtual
protected virtual void Attack() { } // Overridable

// EnemyGoblin.cs
// ✅ Can override - method is virtual
protected override void Attack() { }

4. The Override Keyword: Customizing Behavior 🎨

Concept:

The override keyword allows a child class to provide a new implementation of a virtual method from the parent class. This is where polymorphism happens—you're changing what the method does for a specific child class.

Overriding Attack in EnemyGoblin

EnemyGoblin.cs
using UnityEngine;

public class EnemyGoblin : Enemy
{
    private void StealMoney()
    {
        Debug.Log(enemyName + " steals gold from player!");
    }

    // Override the Attack method from Enemy
    protected override void Attack()
    {
        base.Attack(); // Call parent implementation first
        StealMoney(); // Add unique Goblin behavior
    }
}

🎯 How Override Works

  • protected override: Must match parent's access level and add override keyword
  • base.Attack(): Calls the parent's implementation (Enemy's Attack)
  • StealMoney(): Adds new behavior after parent's behavior
  • Automatic Selection: When Update() calls Attack(), Goblin's version is used
  • Polymorphism in Action: Same method name, different behavior

IntelliSense Shortcut

Easy Override with IntelliSense

- override Attack()
- override TakeDamage()

- override Awake()
- override ToString()

// Select the one you want, IntelliSense generates:

{
    base.Attack();
}

// ✅ Template ready, just add custom code!

5. Understanding base.Method() 🔗

Concept:

The base keyword lets you call the parent class's version of a method. This is crucial when you want to extend behavior (keep the original + add new) rather than replace it entirely.

How base.Method() Works

Execution Flow
// Enemy.cs (base class)
protected virtual void Attack()
{
    Debug.Log(enemyName + " attacks!"); // Step 2: This runs
}

// EnemyGoblin.cs (child class)
protected override void Attack() // Step 1: This is called
{
    base.Attack(); // Step 2: Calls parent's Attack()
    StealMoney(); // Step 3: Then does this
}

FLOW:
1. EnemyGoblin.Attack() called
2. base.Attack() → Enemy.Attack() runs
3. Returns to EnemyGoblin.Attack()
4. StealMoney() runs
5. Method completes

🔑 base Keyword Rules

  • base.Method(): Calls parent's implementation
  • Optional: You don't HAVE to call base (can completely replace)
  • Order Matters: Call base first, middle, or last depending on needs
  • One Level Up: base refers to immediate parent, not grandparent
  • Constructor Use: base() commonly used in constructors (advanced)

Different Orders of base

Controlling Execution Order
// OPTION 1: Base first (most common)
protected override void Attack()
{
    base.Attack(); // Parent behavior first
    StealMoney(); // Then child behavior
}
// Output: "attacks!" → "steals gold!"

// OPTION 2: Base last
protected override void Attack()
{
    StealMoney(); // Child behavior first
    base.Attack(); // Then parent behavior
}
// Output: "steals gold!" → "attacks!"

// OPTION 3: Base in middle
protected override void Attack()
{
    PrepareToSteal();
    base.Attack(); // Parent in middle
    CompleteSteal();
}
// Output: "preparing..." → "attacks!" → "completing..."

With vs Without base

Comparison
// WITH base - EXTENDS parent behavior
protected override void Attack()
{
    base.Attack(); // Keep parent's behavior
    StealMoney(); // Add new behavior
}
// Result: Does BOTH actions

// WITHOUT base - REPLACES parent behavior
protected override void Attack()
{
    StealMoney(); // Only new behavior
}
// Result: Only steals, doesn't attack!

WHEN TO USE:
✅ Use base when extending behavior
❌ Skip base when completely replacing behavior
💡 Design Choice: Whether to call base depends on your design. Goblin should probably attack AND steal (use base). But if you had a PacifistEnemy that doesn't attack at all, you'd override Attack() without calling base!

6. Completely Replacing Methods 🔄

Concept:

Sometimes you don't want to extend the parent's behavior—you want to completely replace it. By overriding a method without calling base.Method(), you define entirely new behavior.

The Archer Example

EnemyArcher.cs - Completely Different Attack
using UnityEngine;

public class EnemyArcher : Enemy
{
    [SerializeField] private GameObject arrowPrefab;
    [SerializeField] private float arrowSpeed = 10f;

    // Override WITHOUT calling base
    protected override void Attack()
    {
        // No base.Attack() - completely new behavior
        Debug.Log(enemyName + " shoots an arrow!");

        // In real project:
        // Instantiate(arrowPrefab, transform.position, Quaternion.identity);
        // arrow.GetComponent().velocity = ...
    }
}

🎯 Archer Attack Behavior

  • No base.Attack(): Parent's melee attack completely ignored
  • New Implementation: Shoots arrow instead of melee
  • Same Method Name: Still called Attack()
  • Polymorphism: Different enemies, different Attack() behaviors
  • Consistent Interface: All enemies have Attack(), easy to loop

Testing Three Enemy Types

Real-World Implementation

Full Archer Attack (Production Code)
public class EnemyArcher : Enemy
{
    [SerializeField] private GameObject arrowPrefab;
    [SerializeField] private Transform shootPoint;
    [SerializeField] private float shootRange = 10f;
    [SerializeField] private float arrowSpeed = 15f;

    protected override void Attack()
    {
        // Check if player in range
        if (CanShoot())
        {
            ShootArrow();
        }
    }

    private bool CanShoot()
    {
        GameObject player = GameObject.FindGameObjectWithTag("Player");
        if (player == null) return false;

        float distance = Vector2.Distance(transform.position, player.transform.position);
        return distance <= shootRange;
    }

    private void ShootArrow()
    {
        Debug.Log(enemyName + " shoots an arrow!");

        // Spawn arrow
        GameObject arrow = Instantiate(arrowPrefab, shootPoint.position, Quaternion.identity);

        // Aim at player
        GameObject player = GameObject.FindGameObjectWithTag("Player");
        Vector2 direction = (player.transform.position - shootPoint.position).normalized;

        // Launch arrow
        Rigidbody2D arrowRb = arrow.GetComponent<Rigidbody2D>();
        arrowRb.velocity = direction * arrowSpeed;
    }
}
🎮 Design Pattern: Ranged enemies (Archer, Mage) typically replace Attack() completely. Melee variations (Goblin, Knight) typically extend Attack() with base. Choose based on whether behavior is similar or completely different!

7. More Polymorphism Examples 🎪

Concept:

Polymorphism isn't just for Attack() methods. Any behavior that varies between enemy types is a candidate for virtual/override. Let's explore common examples from real game development.

Example 1: Die() Method Variations

Different Death Behaviors
// Enemy.cs - Base class
protected virtual void Die()
{
    Debug.Log(enemyName + " dies!");
    DropLoot();
    Destroy(gameObject);
}

// EnemySplitter.cs - Splits into smaller enemies
protected override void Die()
{
    Debug.Log(enemyName + " splits into two!");

    // Spawn two smaller enemies
    Instantiate(smallEnemyPrefab, transform.position + Vector3.left, Quaternion.identity);
    Instantiate(smallEnemyPrefab, transform.position + Vector3.right, Quaternion.identity);

    Destroy(gameObject); // Still dies, but spawns first
}

// EnemyBoss.cs - Epic death sequence
protected override void Die()
{
    base.Die(); // Drop loot like normal enemy

    // Additional boss death effects
    PlayCutscene();
    UnlockNextArea();
    ShowVictoryScreen();
}

Example 2: TakeDamage() Variations

Different Damage Reactions
// Enemy.cs
public virtual void TakeDamage(int damage)
{
    currentHealth -= damage;
    FlashRed();

    if (currentHealth <= 0)
    {
        Die();
    }
}

// EnemyShielded.cs - Has armor that must break first
public override void TakeDamage(int damage)
{
    if (shieldActive)
    {
        shieldHealth -= damage;
        Debug.Log("Shield absorbed damage! " + shieldHealth + " remaining");

        if (shieldHealth <= 0)
        {
            shieldActive = false;
            Debug.Log("Shield broken!");
        }
    }
    else
    {
        base.TakeDamage(damage); // Normal damage after shield breaks
    }
}

// EnemyVampire.cs - Heals nearby enemies when damaged
public override void TakeDamage(int damage)
{
    base.TakeDamage(damage);

    // Heal nearby vampires
    Collider2D[] nearbyEnemies = Physics2D.OverlapCircleAll(transform.position, healRadius);
    foreach (var enemy in nearbyEnemies)
    {
        if (enemy.GetComponent<EnemyVampire>() != null)
        {
            enemy.GetComponent<EnemyVampire>().Heal(5);
        }
    }
}

Example 3: Move() Variations

Different Movement Patterns
// Enemy.cs
protected virtual void Move()
{
    // Simple left-right patrol
    transform.Translate(Vector2.right * moveSpeed * Time.deltaTime);
}

// EnemyFlying.cs - Flies in sine wave
protected override void Move()
{
    float x = Time.time * moveSpeed;
    float y = Mathf.Sin(Time.time * 2) * 2;
    transform.position = startPosition + new Vector3(x, y, 0);
}

// EnemyChaser.cs - Always moves toward player
protected override void Move()
{
    GameObject player = GameObject.FindGameObjectWithTag("Player");
    if (player != null)
    {
        Vector2 direction = (player.transform.position - transform.position).normalized;
        transform.Translate(direction * moveSpeed * Time.deltaTime);
    }
}

// EnemyTeleporter.cs - Teleports instead of walking
protected override void Move()
{
    teleportTimer -= Time.deltaTime;

    if (teleportTimer <= 0)
    {
        Vector3 randomPos = GetRandomPosition();
        transform.position = randomPos;
        teleportTimer = teleportCooldown;
    }
}

🎯 Common Virtual Methods in Games

  • Attack(): Different attack types (melee, ranged, magic)
  • Die(): Different death effects (explode, split, drop loot)
  • TakeDamage(): Different damage reactions (armor, reflect, heal allies)
  • Move(): Different movement patterns (patrol, chase, fly, teleport)
  • OnPlayerDetected(): Different alert behaviors (attack, flee, call allies)
  • Awake()/Start(): Different initialization (load stats, spawn weapons)
🎮 Game Design Tip: Look for behaviors that might vary between enemies and make those methods virtual. Even if you only have one implementation now, future enemy types will thank you!

8. Polymorphism with Loops 🔄

Concept:

The true power of polymorphism shines when working with collections. You can loop through different enemy types and call the same method—each enemy automatically uses its own version!

Looping Through Mixed Enemy Types

Polymorphic Enemy Manager
using UnityEngine;
using System.Collections.Generic;

public class EnemyManager : MonoBehaviour
{
    // List holds ALL enemy types (Skeleton, Goblin, Archer)
    public List<Enemy> allEnemies = new List<Enemy>();

    private void Start()
    {
        // Find all enemies in scene
        allEnemies.AddRange(FindObjectsOfType<Enemy>());
    }

    private void Update()
    {
        // Make ALL enemies attack on F key
        if (Input.GetKeyDown(KeyCode.F))
        {
            MakeAllEnemiesAttack();
        }
    }

    private void MakeAllEnemiesAttack()
    {
        foreach (Enemy enemy in allEnemies)
        {
            enemy.Attack(); // Each uses their own version!
        }
    }
}

// Console output when F pressed:
// Skeleton attacks!
// Goblin attacks!
// Goblin steals gold from player!
// Archer shoots an arrow!

✨ Why This Is Powerful

  • One Loop: Single foreach handles all enemy types
  • Automatic Dispatch: C# automatically calls correct version
  • No If Statements: Don't need to check enemy type manually
  • Easy to Extend: Add new enemy type, loop still works
  • Clean Code: No messy switch statements or type checking

Without Polymorphism (The Ugly Way)

❌ What You'd Need Without Polymorphism
// WITHOUT polymorphism - ugly and unscalable
foreach (GameObject enemyObj in allEnemies)
{
    if (enemyObj.GetComponent<EnemySkeleton>() != null)
    {
        enemyObj.GetComponent<EnemySkeleton>().MeleeAttack();
    }
    else if (enemyObj.GetComponent<EnemyGoblin>() != null)
    {
        enemyObj.GetComponent<EnemyGoblin>().SpecialAttack();
    }
    else if (enemyObj.GetComponent<EnemyArcher>() != null)
    {
        enemyObj.GetComponent<EnemyArcher>().ShootArrow();
    }
    else if (enemyObj.GetComponent<EnemyMage>() != null)
    {
        enemyObj.GetComponent<EnemyMage>().CastSpell();
    }
    // Add 20 more else-if blocks for 20 enemy types...
}

PROBLEMS:
❌ Massive if-else chain
❌ Different method names
❌ Breaks when adding new enemies
❌ Hard to maintain
❌ Not professional

More Loop Examples

Common Polymorphic Patterns
// Kill all enemies
foreach (Enemy enemy in allEnemies)
{
    enemy.TakeDamage(999); // Each handles damage differently
}

// Make all enemies move
foreach (Enemy enemy in allEnemies)
{
    enemy.Move(); // Each moves in their unique pattern
}

// Heal all enemies
foreach (Enemy enemy in allEnemies)
{
    enemy.Heal(20); // Each might have different heal logic
}

// Call custom behavior on all
foreach (Enemy enemy in allEnemies)
{
    enemy.OnPlayerDetected(); // Each reacts differently
}

9. When to Use Virtual/Override 🤔

Concept:

Not every method should be virtual. Overusing virtual/override can make code confusing and slower. Here are guidelines for when polymorphism is the right choice vs when simpler approaches work better.

Good Use Cases for Virtual/Override

✅ Use Polymorphism When:

  • Behavior Varies by Type: Different enemies attack differently
  • Clear Hierarchy: You have a base class with multiple children
  • Shared Interface: Want to call same method on different types
  • Template Method Pattern: Base class defines structure, children fill details
  • Multiple Implementations: Have 3+ different versions of behavior
  • Loop Through Collections: Need to treat different types uniformly

Bad Use Cases (Avoid Virtual/Override)

❌ DON'T Use Polymorphism When:

  • Only One Implementation: No child classes override it
  • Private Helper Methods: Internal methods not meant for overriding
  • Performance Critical: Virtual calls are slightly slower (micro-optimization)
  • Simple Flags Work: If (isBoss) covers it easily
  • Composition Better: Mix-and-match components instead
  • No Clear Hierarchy: Classes aren't really related

Decision Framework

Should This Method Be Virtual?
QUESTION CHECKLIST:

1. "Will child classes need different behavior?"
✅ YES → Consider virtual
❌ NO → Leave non-virtual

2. "Do I have 2+ child classes that would override?"
✅ YES → Make it virtual
❌ NO → Wait until you do

3. "Is this a public/protected interface method?"
✅ YES → Good candidate for virtual
❌ NO (private helper) → Don't make virtual

4. "Can I use a simple bool flag instead?"
✅ YES, and it's simpler → Skip virtual
❌ NO, need real different code → Use virtual

EXAMPLES:

Attack() - Should be virtual?
✅ YES: Different enemies attack differently

GetComponent() - Should be virtual?
❌ NO: Internal helper, not meant to vary

Awake() - Should be virtual?
✅ MAYBE: If children need custom initialization

CalculateDistance() - Should be virtual?
❌ NO: Math is math, doesn't vary by enemy type

Performance Note

Virtual Method Performance
PERFORMANCE IMPACT:

Non-virtual method call:
- Direct jump to method address
- Compiler can inline
- Extremely fast

Virtual method call:
- Look up method in virtual table (vtable)
- Then jump to method address
- Can't inline
- Slightly slower (~1-2 nanoseconds)

REALITY:
- Difference is TINY (nanoseconds)
- Unity has bigger performance concerns (rendering, physics)
- Premature optimization is root of all evil

GUIDELINE:
✅ Use virtual when it makes code cleaner
❌ Only worry about performance in tight loops (Update)
❌ Don't avoid virtual for performance unless profiler proves issue
🎯 Best Practice: Default to non-virtual. When you find yourself needing to override, THEN make it virtual. It's easy to add virtual later, harder to remove it (breaks child classes). Start conservative, add as needed!

10. Advanced: Virtual, Override, and Sealed 🔐

Concept:

C# has additional keywords that control inheritance further: abstract, sealed, and new. Understanding these completes your polymorphism knowledge for advanced scenarios.

The Sealed Keyword

Preventing Further Overrides
// Base class
public class Enemy : MonoBehaviour
{
    protected virtual void Attack() { }
}

// Middle class - overrides and seals
public class EnemyGoblin : Enemy
{
    protected sealed override void Attack()
    {
        base.Attack();
        StealMoney();
    }
}

// Child of Goblin
public class EliteGoblin : EnemyGoblin
{
    // ❌ ERROR: Can't override sealed method
    // protected override void Attack() { }
}

USE CASE:
- Prevent grandchildren from changing behavior
- Lock down critical logic
- Rarely needed in practice

The Abstract Keyword

Forcing Child Classes to Implement
// Abstract base class - can't instantiate
public abstract class Enemy : MonoBehaviour
{
    // Abstract method - NO implementation
    protected abstract void Attack();

    // Can still have normal methods
    public void TakeDamage(int damage)
    {
        // Implementation here
    }
}

// Child MUST implement Attack
public class EnemySkeleton : Enemy
{
    // ✅ REQUIRED: Must override abstract method
    protected override void Attack()
    {
        Debug.Log("Skeleton swings sword!");
    }
}

// ❌ ERROR: Can't create Enemy directly
// Enemy enemy = new Enemy(); // Compile error!

USE CASE:
- Force all children to provide implementation
- Base class is template, not meant to be used directly
- Common in professional architectures

The New Keyword (Hiding, Not Overriding)

Method Hiding
// Base class
public class Enemy : MonoBehaviour
{
    // Not virtual!
    public void Attack()
    {
        Debug.Log("Enemy attacks");
    }
}

// Child class
public class EnemyGoblin : Enemy
{
    // Hides parent method (not polymorphic!)
    public new void Attack()
    {
        Debug.Log("Goblin attacks differently");
    }
}

// Testing
EnemyGoblin goblin = new EnemyGoblin();
goblin.Attack(); // "Goblin attacks differently"

Enemy enemy = goblin; // Upcast to base type
enemy.Attack(); // "Enemy attacks" ← NOT POLYMORPHIC!

DIFFERENCE:
- override: Polymorphic, changes behavior for all references
- new: Hides method, which version runs depends on reference type

WARNING: Avoid 'new' - it's confusing and breaks polymorphism!

Comparison Chart

Keyword Summary
KEYWORD COMPARISON:

virtual:
- Base class method
- CAN be overridden (optional)
- Has implementation

abstract:
- Base class method
- MUST be overridden (required)
- NO implementation

override:
- Child class method
- Replaces virtual/abstract parent
- Polymorphic behavior

sealed override:
- Child class method
- Overrides parent
- Prevents grandchildren from overriding

new:
- Child class method
- Hides parent method
- NOT polymorphic (avoid!)

RECOMMENDATION FOR BEGINNERS:
✅ Use: virtual, override, base
⚠️ Advanced: abstract, sealed
❌ Avoid: new (confusing)
🎓 Learning Path: Master virtual/override first. Abstract and sealed are advanced patterns you'll encounter in large projects. The 'new' keyword for hiding is generally considered bad practice—prefer proper polymorphism with virtual/override!

📚 Lesson 12 Summary

Outstanding work! You've mastered the second pillar of Object-Oriented Programming. Here's everything you learned:

  • ✅ Polymorphism Definition: One interface, many implementations—same method, different behaviors
  • ✅ Real-World Analogy: "Dance" command—everyone dances, but in their own unique way
  • ✅ Virtual Keyword: Makes methods overridable by child classes
  • ✅ Override Keyword: Replaces parent method with child-specific implementation
  • ✅ base.Method(): Calls parent's implementation—use to extend behavior
  • ✅ Extending Behavior: Goblin.Attack() calls base + adds StealMoney()
  • ✅ Replacing Behavior: Archer.Attack() shoots arrow without calling base
  • ✅ Polymorphic Loops: foreach (Enemy e in enemies) e.Attack() works for all types
  • ✅ Common Virtual Methods: Attack(), Die(), TakeDamage(), Move() all benefit from polymorphism
  • ✅ When to Use: Behavior varies by type, clear hierarchy, shared interface needed
  • ✅ Advanced Keywords: Sealed (prevent override), abstract (force override), new (avoid!)
  • ✅ Professional Pattern: Treat different types uniformly while preserving unique behaviors
