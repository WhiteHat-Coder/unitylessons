1. What is Object-Oriented Programming? 🏛️

Concept:

Object-Oriented Programming (OOP) is a programming paradigm based on organizing code into reusable "objects" that contain data and behaviors. OOP has three main pillars: Inheritance, Polymorphism, and Encapsulation. Mastering these principles transforms messy code into clean, maintainable systems.

🎮 Three Pillars of OOP

  • Inheritance: Share code between classes through parent-child relationships
  • Polymorphism: One interface, multiple implementations (covered in next lesson)
  • Encapsulation: Hide internal details, expose only what's needed
💡 Philosophy: OOP principles are "easy to learn, hard to master." The concepts are simple, but knowing when and how to apply them separates junior from senior developers. Don't worry—we'll practice extensively!

Why OOP Matters in Unity

The Problem Without OOP
SCENARIO: You have 5 enemy types

Without OOP:
- EnemySkeleton script: 300 lines
- EnemyGoblin script: 320 lines
- EnemyArcher script: 310 lines
- EnemyMage script: 330 lines
- EnemyBoss script: 400 lines

Total: 1,660 lines
Problem: 80% of code is DUPLICATED!

With OOP:
- Enemy base class: 200 lines (shared)
- EnemySkeleton: 50 lines (unique)
- EnemyGoblin: 60 lines (unique)
- EnemyArcher: 55 lines (unique)
- EnemyMage: 70 lines (unique)
- EnemyBoss: 100 lines (unique)

Total: 535 lines (67% reduction!)

✨ Benefits of OOP

  • Code Reusability: Write once, use everywhere
  • Easier Maintenance: Fix bugs in one place, not five
  • Scalability: Add new enemies without rewriting everything
  • Organization: Clear structure makes teamwork easier
  • Professional Standard: Industry expects OOP knowledge

2. Understanding Inheritance 🌳

Concept:

Inheritance allows one class to inherit properties and functionalities from another class. Think of it as a parent-child relationship: the parent (base class) contains general functionality, and children (derived classes) inherit that functionality while adding their own unique features.

Real-World Analogy

Family Tree Metaphor
PARENT: Vehicle
- Has wheels
- Has engine
- Can accelerate
- Can brake

CHILD: Car (inherits from Vehicle)
- ✅ Has wheels (inherited)
- ✅ Has engine (inherited)
- ✅ Can accelerate (inherited)
- ✅ Can brake (inherited)
- ➕ Has 4 doors (unique)
- ➕ Can play radio (unique)

CHILD: Motorcycle (inherits from Vehicle)
- ✅ Has wheels (inherited)
- ✅ Has engine (inherited)
- ✅ Can accelerate (inherited)
- ✅ Can brake (inherited)
- ➕ Has 2 wheels (unique)
- ➕ Can do wheelie (unique)

🎯 Inheritance Terminology

  • Base Class (Parent): Contains shared functionality
  • Derived Class (Child): Inherits from parent + adds unique features
  • Inherits: Receives functionality from parent
  • Extends: Adds new functionality on top of inherited features

Inheritance in Unity

Unity Class Hierarchy
// You've been using inheritance all along!
public class PlayerController : MonoBehaviour
{
    // Your code uses Start(), Update(), transform, gameObject
    // All inherited from MonoBehaviour!
}

Hierarchy:
Object (C# base class)
  ↓
MonoBehaviour (Unity class)
  ↓
PlayerController (Your class)

PlayerController inherits from MonoBehaviour,
which inherits from Object.
🎓 Revelation: Every Unity script you've written uses inheritance! When you type : MonoBehaviour, you're inheriting Unity's built-in functionality like Update(), Start(), and GetComponent().

3. Creating a Base Enemy Class 👾

Concept:

Let's build a practical example: an Enemy base class with functionality shared by all enemies. Every enemy can walk, has a name, and can attack. This shared code belongs in the base class.

Identifying Shared Functionality

🧩 What Do ALL Enemies Have?

  • Move Speed: Every enemy moves at some speed
  • Name: Every enemy has a name (Skeleton, Goblin, etc.)
  • Walk Behavior: Every enemy can move around
  • Attack Behavior: Every enemy can attack the player
  • Health (optional): Every enemy has HP
  • Damage (optional): Every enemy deals damage

Creating Enemy.cs Script

Base Enemy Class
using UnityEngine;

public class Enemy : MonoBehaviour
{
    // Variables shared by ALL enemies
    public float moveSpeed;
    public string enemyName;

    // Method shared by ALL enemies
    private void MoveAround()
    {
        Debug.Log(enemyName + " moves at speed " + moveSpeed);
    }

    // Another shared method
    private void Attack()
    {
        Debug.Log(enemyName + " attacks!");
    }

    private void Update()
    {
        MoveAround(); // Always moving

        // Attack on F key press
        if (Input.GetKeyDown(KeyCode.F))
        {
            Attack();
        }
    }
}

🎯 Code Breakdown

  • public float moveSpeed: Visible in Inspector, each enemy has different speed
  • public string enemyName: Set per enemy in Inspector
  • MoveAround(): Simple debug message showing movement
  • Attack(): Simple debug message showing attack
  • Update(): Calls MoveAround() every frame, Attack() on F key
📝 Simplified Example: We're using Debug.Log for simplicity. In real projects, these methods would contain animations, physics movement, damage calculations, cooldowns, etc. Focus on the inheritance concept first!

4. Creating a Child Class (EnemyGoblin) 👺

Concept:

Now let's create EnemyGoblin—a child class that inherits all Enemy functionality but also has unique abilities. Goblins can do everything enemies can do, PLUS steal gold from the player.

The Wrong Way (Copy-Paste)

❌ BAD: Code Duplication
// DON'T DO THIS!
public class EnemyGoblin : MonoBehaviour
{
    // Copied from Enemy class
    public float moveSpeed;
    public string enemyName;

    private void MoveAround() { /* copied code */ }
    private void Attack() { /* copied code */ }
    public void TakeDamage() { /* copied code */ }

    // New goblin-specific method
    private void StealMoney() { /* new code */ }
}

PROBLEMS:
- Code duplication (messy!)
- Bug in Enemy? Fix in 5 places!
- Hard to maintain
- Not scalable

The Right Way (Inheritance)

✅ GOOD: Using Inheritance
// Inherit from Enemy, not MonoBehaviour!
public class EnemyGoblin : Enemy
{
    // NO variables needed - inherited from Enemy!
    // NO MoveAround() needed - inherited!
    // NO Attack() needed - inherited!

    // Only add NEW functionality
    private void StealMoney()
    {
        Debug.Log(enemyName + " steals gold from player!");
    }

    [ContextMenu("Steal Gold")]
    private void StealGold_Test()
    {
        StealMoney();
    }
}

🎯 How Inheritance Works Here

  • : Enemy means "inherit from Enemy class"
  • moveSpeed, enemyName: Available automatically (inherited)
  • MoveAround(), Attack(): Available automatically (inherited)
  • StealMoney(): New method, only Goblins have this
  • Enemy → MonoBehaviour: Goblin inherits from Enemy, which inherits from MonoBehaviour

Inheritance Chain

Class Hierarchy
MonoBehaviour (Unity base class)
  ↓ inherits
Enemy (your base class)
- moveSpeed
- enemyName
- MoveAround()
- Attack()
- TakeDamage()
  ↓ inherits
EnemyGoblin (your child class)
- ✅ moveSpeed (inherited)
- ✅ enemyName (inherited)
- ✅ MoveAround() (inherited)
- ✅ Attack() (inherited)
- ✅ TakeDamage() (inherited)
- ✅ Update() (inherited)
- ✅ Start(), Awake() (inherited from MonoBehaviour)
- ➕ StealMoney() (unique to Goblin)
🎓 Key Insight: Even though EnemyGoblin.cs looks empty except for StealMoney(), it has ALL the functionality of Enemy AND MonoBehaviour. That's the power of inheritance!

5. Access Modifiers: Public, Private, Protected 🔒

Concept:

Access modifiers control who can see and use your variables and methods. There are three main types: public (everyone can access), private (only this class can access), and protected (this class + child classes can access).

Understanding Access Levels

Access Modifier Comparison
public class Enemy : MonoBehaviour
{
    // PUBLIC: Anyone can access
    public float moveSpeed;
    // ✅ Visible in Inspector
    // ✅ Other scripts can access: enemy.moveSpeed
    // ✅ Child classes can access

    // PRIVATE: Only Enemy class can access
    private string enemyName;
    // ❌ Hidden in Inspector
    // ❌ Other scripts CANNOT access
    // ❌ Child classes CANNOT access

    // PROTECTED: This class + children can access
    protected int health;
    // ❌ Hidden in Inspector (unless SerializeField)
    // ❌ Other scripts CANNOT access
    // ✅ Child classes CAN access
}

🔑 Access Modifier Rules

  • public: Accessible everywhere (Inspector, other scripts, child classes)
  • private: Only accessible in the class where it's declared
  • protected: Accessible in declaring class + child classes
  • Default: If you don't specify, it's private by default

The Problem with Public

Why Not Make Everything Public?
// BAD: Everything public
public class Enemy : MonoBehaviour
{
    public float moveSpeed;
    public string enemyName;
    public int currentHealth;
    public bool isDead;
    public float attackTimer;
}

PROBLEMS:
- Any script can change any value: enemy.isDead = true
- No control over how data is modified
- Hard to track bugs (who changed what?)
- Cluttered Inspector with 20+ fields
- Not professional practice
💼 Professional Rule: Make everything private by default, then selectively make things public/protected only when needed. This is called encapsulation—one of the OOP pillars!

The Problem with Private for Inheritance

Private Variables Break Inheritance
// In Enemy.cs
public class Enemy : MonoBehaviour
{
    private float moveSpeed;
}

// In EnemyGoblin.cs
public class EnemyGoblin : Enemy
{
    private void Awake()
    {
        moveSpeed = 5f; // ❌ ERROR!
        // "moveSpeed is inaccessible due to its protection level"
    }
}

The Solution: Protected

✅ Protected Allows Inheritance
// In Enemy.cs
public class Enemy : MonoBehaviour
{
    protected float moveSpeed; // Protected instead of private
}

// In EnemyGoblin.cs
public class EnemyGoblin : Enemy
{
    private void Awake()
    {
        moveSpeed = 5f; // ✅ Works! Protected allows child access
        Debug.Log(enemyName); // ✅ Works if enemyName is also protected
    }
}

✨ Protected Benefits

  • Private to Outside: Other scripts can't access protected members
  • Accessible to Children: Child classes can read/modify protected members
  • Better Than Public: More controlled than public access
  • Better Than Private: Allows inheritance without full exposure
  • Professional Standard: Used extensively in real game projects

6. Combining [SerializeField] with Protected 🎛️

Concept:

There's a problem: protected variables are hidden in the Inspector. We want Inspector visibility for designers to tune values, but we also want protection from outside scripts. The solution: [SerializeField] + protected!

The Inspector Problem

Protected Hides from Inspector
public class Enemy : MonoBehaviour
{
    protected float moveSpeed = 3f;
    protected string enemyName = "Enemy";
}

RESULT IN INSPECTOR:
❌ moveSpeed not visible
❌ enemyName not visible
❌ Can't tune values per enemy

PROBLEM:
- Every enemy stuck with same moveSpeed (3f)
- Can't customize in Inspector
- Designers can't adjust without editing code

The SerializeField Solution

✅ Best of Both Worlds
public class Enemy : MonoBehaviour
{
    [SerializeField] protected float moveSpeed = 3f;
    [SerializeField] protected string enemyName = "Enemy";
}

RESULT:
✅ Visible in Inspector (designers can tune)
✅ Protected from outside scripts (other scripts can't access)
✅ Accessible in child classes (EnemyGoblin can use/modify)
✅ Professional and flexible

🎯 [SerializeField] Behavior

  • Makes Private/Protected Visible: Shows in Inspector despite access modifier
  • Doesn't Change Access: Still protected, just serialized for Unity
  • Unity-Specific: Only works in Unity Engine
  • Common Pattern: Used extensively in professional Unity projects

Complete Base Class with SerializeField

Professional Enemy Base Class
using UnityEngine;

public class Enemy : MonoBehaviour
{
    [SerializeField] protected float moveSpeed = 3f;
    [SerializeField] protected string enemyName = "Enemy";
    [SerializeField] protected int maxHealth = 100;

    protected int currentHealth;

    protected virtual void Awake()
    {
        currentHealth = maxHealth;
    }

    protected void MoveAround()
    {
        Debug.Log(enemyName + " moves at speed " + moveSpeed);
    }

    protected void Attack()
    {
        Debug.Log(enemyName + " attacks!");
    }

    public void TakeDamage(int damage)
    {
        currentHealth -= damage;
        Debug.Log(enemyName + " takes " + damage + " damage!");

        if (currentHealth <= 0)
        {
            Die();
        }
    }

    protected void Die()
    {
        Debug.Log(enemyName + " dies!");
        // Play death animation, drop loot, etc.
    }

    protected void Update()
    {
        MoveAround();

        if (Input.GetKeyDown(KeyCode.F))
        {
            Attack();
        }
    }
}
🎨 Design Note: Notice methods are also protected. This lets child classes call or modify these methods (polymorphism—next lesson topic!). Public methods like TakeDamage() are interfaces for outside scripts to interact with enemies.

7. Adding Unique Child Functionality 🎭

Concept:

Child classes shine when they add unique functionality that parent doesn't have. EnemyGoblin inherits all Enemy features, but ONLY Goblins can steal gold. This is how you build diverse enemy types without code duplication.

Goblin-Specific Behavior

EnemyGoblin with Unique Ability
using UnityEngine;

public class EnemyGoblin : Enemy
{
    [SerializeField] private int goldStealAmount = 10;

    // Unique method only Goblins have
    private void StealMoney()
    {
        // Can use enemyName even though not declared here
        // because it's inherited (protected) from Enemy
        Debug.Log(enemyName + " steals " + goldStealAmount + " gold from player!");

        // In real game: PlayerInventory.gold -= goldStealAmount;
    }

    [ContextMenu("Steal Gold")]
    private void StealGold_Menu()
    {
        StealMoney();
    }

    // Can also access inherited methods
    private void OnTriggerEnter2D(Collider2D collision)
    {
        if (collision.CompareTag("Player"))
        {
            StealMoney();
            Attack(); // Inherited from Enemy
        }
    }
}

🎯 What EnemyGoblin Has

  • ✅ moveSpeed: Inherited from Enemy
  • ✅ enemyName: Inherited from Enemy
  • ✅ maxHealth, currentHealth: Inherited from Enemy
  • ✅ MoveAround(): Inherited from Enemy
  • ✅ Attack(): Inherited from Enemy
  • ✅ TakeDamage(): Inherited from Enemy
  • ✅ Die(): Inherited from Enemy
  • ➕ StealMoney(): UNIQUE to EnemyGoblin
  • ➕ goldStealAmount: UNIQUE to EnemyGoblin

Multiple Child Classes

Creating More Enemy Types
// EnemyArcher.cs - Ranged enemy
public class EnemyArcher : Enemy
{
    [SerializeField] private GameObject arrowPrefab;
    [SerializeField] private float shootRange = 10f;

    private void ShootArrow()
    {
        Debug.Log(enemyName + " shoots an arrow!");
        // Instantiate arrow, aim at player, etc.
    }
}

// EnemyMage.cs - Magic caster
public class EnemyMage : Enemy
{
    [SerializeField] private float mana = 100f;
    [SerializeField] private float spellCooldown = 3f;

    private void CastFireball()
    {
        if (mana >= 20)
        {
            mana -= 20;
            Debug.Log(enemyName + " casts Fireball!");
        }
    }
}

// EnemyBoss.cs - Powerful boss
public class EnemyBoss : Enemy
{
    [SerializeField] private int phase = 1;

    private void EnterPhase2()
    {
        phase = 2;
        moveSpeed *= 1.5f; // Speed up!
        Debug.Log(enemyName + " enters Phase 2!");
    }
}
🎮 Scalability: With one 50-line Enemy base class, you can create 20 different enemy types by writing 20-30 lines each. Without inheritance, each enemy would be 200+ lines of duplicated code!

8. Real-World Example: Entity Base Class 🎪

Concept:

In professional Unity projects, you often create an even higher base class called Entity that covers ALL characters—players, enemies, NPCs. This maximizes code reuse across your entire game.

Entity Base Class Pattern

Professional Architecture
HIERARCHY:

Entity (base for ALL characters)
├─ Player (inherits from Entity)
├─ Enemy (inherits from Entity)
│ ├─ EnemySkeleton (inherits from Enemy)
│ ├─ EnemyGoblin (inherits from Enemy)
│ ├─ EnemyArcher (inherits from Enemy)
│ └─ EnemyBoss (inherits from Enemy)
└─ NPC (inherits from Entity)
├─ Shopkeeper (inherits from NPC)
└─ QuestGiver (inherits from NPC)

SHARED IN ENTITY:
- Health system
- Animator component reference
- Sprite flipping
- Collision detection
- Ground checking
- Movement physics

Entity Base Class Example

Entity.cs
using UnityEngine;

public class Entity : MonoBehaviour
{
    // Components ALL entities have
    [SerializeField] protected Animator animator;
    [SerializeField] protected Rigidbody2D rb;
    [SerializeField] protected SpriteRenderer sr;

    // Stats ALL entities have
    [SerializeField] protected int maxHealth = 100;
    protected int currentHealth;

    // Ground detection (from Lesson 7)
    [SerializeField] protected Transform groundCheck;
    [SerializeField] protected float groundCheckDistance;
    [SerializeField] protected LayerMask groundLayer;

    protected bool isGrounded;

    protected virtual void Awake()
    {
        // Get components (shared by all)
        animator = GetComponent<Animator>();
        rb = GetComponent<Rigidbody2D>();
        sr = GetComponent<SpriteRenderer>();

        currentHealth = maxHealth;
    }

    // Flip method (from Lesson 6)
    protected void Flip()
    {
        Vector3 scale = transform.localScale;
        scale.x *= -1;
        transform.localScale = scale;
    }

    // Ground check (from Lesson 7)
    protected void CheckGround()
    {
        isGrounded = Physics2D.Raycast(
            groundCheck.position,
            Vector2.down,
            groundCheckDistance,
            groundLayer
        );
    }

    // Health system (all can take damage)
    public virtual void TakeDamage(int damage)
    {
        currentHealth -= damage;

        if (currentHealth <= 0)
        {
            Die();
        }
    }

    protected virtual void Die()
    {
        // Base death logic
    }
}

🎯 Why Entity Base Class?

  • Player Needs It: Health, animator, ground check, flipping
  • Enemies Need It: Health, animator, ground check, flipping
  • NPCs Need It: Health, animator, ground check (maybe not flipping)
  • Write Once: GetComponent logic in one place
  • Fix Once: Bug in ground check? Fix in Entity, affects all characters

Then Enemy Inherits from Entity

Enemy.cs (Simplified)
public class Enemy : Entity // Inherits from Entity, not MonoBehaviour
{
    // Enemy-specific stuff
    [SerializeField] protected float moveSpeed;
    [SerializeField] protected int damage;

    protected void Attack()
    {
        // Enemy attack logic
    }

    // Still has ALL Entity functionality:
    // - animator, rb, sr
    // - maxHealth, currentHealth
    // - Flip(), CheckGround(), TakeDamage(), Die()
}
🏗️ Architecture Insight: Professional Unity projects often have 3-4 levels of inheritance: MonoBehaviour → Entity → Enemy → EnemySkeleton. Each level adds more specific functionality. This creates clean, maintainable code.

9. When NOT to Use Inheritance ⚠️

Concept:

Inheritance is powerful, but it's not always the answer. Sometimes composition (using components) is better than inheritance. Knowing when to use each approach is a senior-level skill.

Inheritance Problems

The Gorilla-Banana Problem
FAMOUS QUOTE:
"The problem with object-oriented languages is they've
got all this implicit environment that they carry around
with them. You wanted a banana but what you got was a
gorilla holding the banana and the entire jungle."
- Joe Armstrong

TRANSLATION:
- You need ONE method from a class
- You inherit from that class
- Now you have 50 methods and 20 variables
- Your child class is bloated with stuff it doesn't need

When Inheritance Gets Messy

The Diamond Problem
FlyingEnemy
├─ Can fly
└─ Can attack

SwimmingEnemy
├─ Can swim
└─ Can attack

QUESTION: What if you want FlyingSwimmingEnemy?

❌ Can't inherit from both in C#
❌ Copy-paste code from both? (code duplication)
❌ Make FlyingSwimmingEnemy base class? (doesn't make sense)

SOLUTION: Use composition (components) instead!

Composition Alternative

Component-Based Design
// Instead of inheritance, use components:

public class FlyingComponent : MonoBehaviour
{
    public void Fly() { /* flying logic */ }
}

public class SwimmingComponent : MonoBehaviour
{
    public void Swim() { /* swimming logic */ }
}

public class AttackComponent : MonoBehaviour
{
    public void Attack() { /* attack logic */ }
}

// Mix and match components:
FlyingEnemy: FlyingComponent + AttackComponent
SwimmingEnemy: SwimmingComponent + AttackComponent
FlyingSwimmingEnemy: FlyingComponent + SwimmingComponent + AttackComponent

Flexible and no inheritance needed!

🎯 Inheritance vs Composition

  • Use Inheritance When: Clear "is-a" relationship (Goblin IS-A Enemy)
  • Use Inheritance When: Child needs most/all parent functionality
  • Use Inheritance When: Hierarchy is shallow (2-3 levels max)
  • Use Composition When: "Has-a" relationship (Enemy HAS-A flying ability)
  • Use Composition When: Need mix-and-match behaviors
  • Use Composition When: Deep inheritance trees (4+ levels)
💼 Professional Balance: Real projects use BOTH. Inheritance for core character hierarchies (Entity → Player/Enemy), composition for abilities and behaviors (flying, swimming, invisible). Master both approaches!

10. Inheritance Best Practices 🎯

Concept:

Following best practices ensures your inheritance hierarchies stay clean, maintainable, and scalable. Here are the rules professionals follow when designing class hierarchies.

Rule 1: Keep Hierarchies Shallow

Depth Guidelines
✅ GOOD: Shallow (2-3 levels)
MonoBehaviour → Enemy → EnemyGoblin

⚠️ ACCEPTABLE: Medium (3-4 levels)
MonoBehaviour → Entity → Enemy → EnemyMelee → EnemySkeleton

❌ BAD: Deep (5+ levels)
MonoBehaviour → Entity → LivingEntity → Enemy →
MeleeEnemy → HumanoidEnemy → SkeletonEnemy → EliteSkeleton

WHY DEEP IS BAD:
- Hard to understand
- Hard to debug
- Hard to modify
- Tight coupling

Rule 2: Use Protected for Shared Members

Access Modifier Strategy
public class Enemy : MonoBehaviour
{
    // ✅ GOOD: Protected for child access
    [SerializeField] protected float moveSpeed;
    [SerializeField] protected int maxHealth;

    // ✅ GOOD: Public for outside interface
    public void TakeDamage(int damage) { }

    // ✅ GOOD: Protected for inheritance
    protected void Die() { }

    // ✅ GOOD: Private for internal only
    private void CalculatePathfinding() { }
}

GUIDELINE:
- Public: External interface (TakeDamage, GetHealth)
- Protected: Shared with children (Die, Attack)
- Private: Internal implementation (helper methods)

Rule 3: Favor Composition Over Inheritance

🎯 Decision Framework

  • Ask "Is-A": If Goblin IS-A Enemy → Use inheritance
  • Ask "Has-A": If Enemy HAS-A flying ability → Use composition
  • Ask "Needs Most": Does child need 80%+ of parent? → Inheritance
  • Ask "Mix-Match": Need different combinations? → Composition

Rule 4: Don't Inherit to Reuse One Method

Wrong Reason to Inherit
// ❌ BAD: Inheriting for one utility method
public class MathUtils : MonoBehaviour
{
    protected float CalculateDistance(Vector3 a, Vector3 b)
    {
        return Vector3.Distance(a, b);
    }
}

public class Enemy : MathUtils // ❌ Wrong!
{
    // Now Enemy is tied to MathUtils for one method
}

// ✅ BETTER: Static utility class
public static class MathUtils
{
    public static float CalculateDistance(Vector3 a, Vector3 b)
    {
        return Vector3.Distance(a, b);
    }
}

// Use it anywhere without inheritance
float distance = MathUtils.CalculateDistance(pos1, pos2);

Rule 5: Document Your Hierarchy

Clear Comments
/// <summary>
/// Base class for all enemy types.
/// Contains shared movement, attack, and health systems.
/// Inherit from this for new enemy types.
/// </summary>
public class Enemy : Entity
{
    // Clear structure
}

/// <summary>
/// Goblin enemy with unique gold-stealing ability.
/// Inherits standard enemy behavior from Enemy class.

    // Goblin-specific code
}
